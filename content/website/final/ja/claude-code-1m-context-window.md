# Claude Codeが5倍に：100万トークンのコンテキストウィンドウがビルダーの常識を変える

金曜の午後。Claude Code v2.1.75がリリースされた。チェンジログを開いて二度見した。

**100万トークンのコンテキストウィンドウ。従来の5倍。価格は同じ。**

Claude Codeで本格的に開発してきた人なら、すべてのセッションを規定していた制約を知っているはずだ。コンテキスト圧縮。エージェントが作業内容を忘れ始める見えない天井。アーキテクチャを一から説明し直さなければならない瞬間。長いセッションが混乱に劣化していくあの感覚。その制約が、一気に吹き飛んだ。

## 実際に何が変わったのか

数字はシンプルだ：

- **以前：** 約20万トークンのコンテキストウィンドウ
- **今：** 100万トークンのコンテキストウィンドウ
- **価格差：** ゼロ。同じMaxサブスクリプション。

しかし、その意味するところはまったくシンプルではない。

## なぜこれがまったく異なる働き方なのか

20万トークンの実際の体感はこうだった。コードベースを探索し、アプローチを計画し、機能を実装し、たぶんテストを実行するところまでで限界が来る。複数ファイルにまたがる複雑なリファクタリングでは、ファイルを読むだけでコンテキストを消費してしまう。フロントエンド、バックエンド、データベース、デプロイメントにまたがるフルスタック機能の開発では、常に圧縮への不安を抱えていた。

100万トークンでは、その計算が根本的に変わる。

**セッションが脆くなくなった。** 以前はコンテキスト制限を中心に作業を構造化していた。タスクをより小さなチャンクに分割し、セッション間でハンドオフドキュメントを書き、圧縮カウンターを常に監視していた。今は、機能の開発を始めて、問題に直面し、代替案を検討し、リファクタリングし、テストし、エッジケースを修正し、デプロイするまで、すべてを一つの連続したセッションで行える。エージェントは話の流れを見失わない。

**コードベース全体がワーキングメモリに収まる。** 私のモノレポには3つのNext.jsアプリ、共有パッケージ、3言語にわたるi18n、GTMオペレーションレイヤー、自動化スクリプトがある。以前は、エージェントが一度に保持できるのはせいぜい1つのアプリのコンテキストだった。今はシステム全体を横断して推論できる。すべてに影響する横断的な変更が、複数セッションのマラソンから単一セッションのスプリントに変わる。

**複雑なデバッグが実際に機能する。** 最も厄介なバグは交差点に潜んでいる。ミドルウェアとi18n設定とデプロイメントパイプラインの相互作用。それらのデバッグには、大量のコンテキストを同時に保持する必要がある。20万トークンでは、深掘りするにつれて以前の発見が失われていた。100万トークンでは、ブラウザコンソールからミドルウェアスタックを通り、ビルド設定を経て戻るまで、バグを一気に追跡できる。エージェントは3ファイル前に見つけたことを覚えている。

**もう監視する必要がない。** 権限の自動承認はすでに設定済みだった。コンテキスト制限は、最後に残った自分で積極的に管理しなければならないものだった。今は野心的なタスクを渡して、本当にその場を離れられる。混乱して重要な情報を圧縮してしまったエージェントではなく、完成した成果物に戻ってこられる。

## 週末ビルドにとっての意味

金曜日だ。つまり、平日のスロットに収まらないビルドの時間だ。野心的なもの。「もしかしてこれ、できるんじゃないか...」で始まるもの。

20万トークンのコンテキストでは、そうしたプロジェクトには慎重なセッション管理が必要だった。フェーズに分ける。ハンドオフドキュメントを書く。セッションごとにエージェントの方向を再設定する。オーバーヘッドのせいで勢いを失う。

100万トークンのコンテキストでは、土曜の朝に座って「仕様からデプロイまで丸ごと一つの機能を作ってくれ」と言い、エージェントが最初から最後まで一貫性を保つことに本当の確信が持てる。モデルが賢くなったわけではない。同じOpus 4.6だ。ただ、自分自身の以前の作業を読み直すことにキャパシティを費やすのではなく、本当の問題にその知性を使えるだけのコンテキストをようやく保持できるようになったのだ。

## 複利効果

これは単なる5倍のトークンではない。1セッションあたりで可能なことの質的な転換だ：

- 依存関係グラフを見失わずに20以上のファイルに触れるリファクタリング
- データベーススキーマからAPI、UI、テストまで通した完全な機能開発
- 単一セッションでの複数コードベース操作
- アプローチを決定する前の深いアーキテクチャ探索 - 探索自体でコンテキストを消費することなく
- エージェントがセッション中の以前のミスを覚えているため、自ら軌道修正するより長い自律実行

これらはすべて、以前も理論上は可能だった。実際には、コンテキスト制限により、うまくいっても不安定、最悪の場合は不可能だった。

## ビルダーのアドバンテージ

Claude Codeで実際のプロダクトを出荷しているなら - AIについてブログ記事を書くのではなく、デモ動画を作るのではなく、本当に出荷しているなら - このアップデートはツールがローンチされて以来、最大のQOL改善だ。

この空間を効果的に使う方法を見つけたビルダーは、急速にリードを広げるだろう。他の人にないアクセスがあるからではない（同じサブスクリプションだ）。今可能になったことを中心にワークフローを再構築するからだ。セッションあたりのタスクが大きくなる。オーバーヘッドが減る。出荷が増える。

同じOpus 4.6。同じ価格。同じツール。ただ、実際に使うための余裕が5倍になった。

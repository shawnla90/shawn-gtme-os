# Claude Code 刚刚升级5倍：100万上下文窗口改变了构建者的一切

周五下午。Claude Code v2.1.75 发布。我打开更新日志，看了两遍。

**100万令牌上下文窗口。之前的5倍。价格不变。**

如果你一直在认真用 Claude Code 构建，你已经知道定义每次会话的那个约束：上下文压缩。那个看不见的天花板，智能体开始忘记它在做什么，你不得不重新解释架构，长会话退化为混乱。那个约束刚刚被彻底打破。

## 实际改变了什么

数字很简单：

- **之前：** ~20万上下文窗口
- **现在：** 100万上下文窗口
- **价格差异：** 零。同样的 Max 订阅。

影响一点都不简单。

## 为什么这是一种不同的工作方式

20万在实践中是这样的：你可以探索代码库，规划方法，实现一个功能，也许在空间紧张之前运行测试。复杂的多文件重构？你光读文件就会烧掉上下文。涉及前端、后端、数据库和部署的全栈功能？你整个过程都在管理压缩焦虑。

在100万时，这个数学从根本上改变了。

**会话不再脆弱了。** 我以前围绕上下文限制来安排工作。把任务拆成小块，在会话之间写交接文档，盯着压缩计数器。现在我可以开始一个功能，遇到复杂情况，探索替代方案，重构，测试，修复边缘情况，然后部署。全在一个连续的会话里。智能体不会丢失思路。

**整个代码库都能放进工作记忆。** 我的单仓库有3个 Next.js 应用、共享包、3种语言的国际化、一个 GTM 运营层和自动化脚本。之前，智能体一次最多能保持一个应用的上下文。现在它能在整个系统中推理。涉及所有内容的横切变更从多会话马拉松变成了单会话冲刺。

**复杂调试真正有效了。** 最难的 bug 存在于交叉点：中间件与国际化配置交互，与部署管道交互。调试这些需要同时持有大量上下文。在20万时，你在深入探索时会丢失之前的发现。在100万时，你可以从浏览器控制台追踪一个 bug，穿过中间件栈，穿过构建配置，然后回来。智能体记得三个文件前发现的东西。

**不用再看管了。** 我已经开启了权限自动批准。上下文限制是我最后需要主动管理的东西。现在我可以交出有野心的任务然后真正走开。回来看到完成的工作，而不是一个压缩掉重要信息的困惑智能体。

## 这对周末项目意味着什么

今天是周五。这意味着是时候做那些工作日放不下的项目了。有野心的那些。以"我想知道能不能..."开头的那些。

在20万上下文时，这些项目需要精心的会话管理。拆分成阶段。写交接文档。每次会话重新定位智能体。在管理开销中失去动力。

在100万上下文时，我可以周六早上坐下来说"从规格到部署给我建一整个功能"，并真正有信心智能体会从头到尾保持连贯。不是因为模型变聪明了。同样的 Opus 4.6。而是因为它终于能持有足够的上下文，把智能用在真实问题上，而不是把能力花在重新阅读自己之前的工作上。

## 复利效应

这不只是多了5倍令牌。这是每次会话可能性的质变：

- 涉及20+文件的重构而不丢失依赖图
- 从数据库 schema 到 API 到 UI 到测试的完整功能开发
- 单次会话中的多代码库操作
- 在确定方法前的深度架构探索，不会在探索本身上烧掉上下文
- 更长的自主运行，智能体能自我纠正，因为它记得会话早期的错误

这些在以前理论上都是可能的。实践中，上下文限制使它们最好时脆弱，最坏时不可能。

## 构建者优势

如果你正在用 Claude Code 发布真实产品...不是写关于 AI 的博客，不是做演示视频，是真正在发布...这次更新是自工具发布以来最大的生活质量改进。

那些想出如何有效使用这个空间的构建者会快速拉开差距。不是因为他们有别人没有的访问权（同一个订阅），而是因为他们会围绕现在可能的事情重构工作流。每次会话更大的任务。更少的管理开销。更多发布。

同样的 Opus 4.6。同样的价格。同样的工具。只是多了5倍的空间来真正使用它。

周五快乐。是时候构建了。

here's what actually works better: haiku 4.5 with a well-structured skill layer.

**what "skill layer" means in practice**

a skill is just a markdown file your model loads before it does the task. it's not a system prompt. it's not RAG. it's a pre-baked decision tree + output contract for a specific operation.

for enrichment workflows, each skill answers three questions:
- what input am I getting (company domain, LinkedIn URL, job title string, etc.)
- what decision logic should I apply before touching the API
- what exact format do I return

that's it. no elaborate prompt chaining. no tool-calling loops. one call, one answer.

**the clay setup I'm running**

I have a skills/ folder in my repo. each file is named for the operation:

```
skills/
 enrich-company-hq.md
 score-persona-fit.md
 classify-tech-stack.md
 extract-buying-signal.md
```

inside `score-persona-fit.md` it looks roughly like:

```
# persona fit scorer

## input contract
- job_title: string
- seniority_level: string (inferred from title)
- department: string

## scoring rubric
- CTO / VP Eng / Head of Eng at <500 employees = 5
- Director of Engineering = 4
- Senior engineer with "platform" or "infra" in title = 3
- IC engineer = 2
- anything else = 1

## output contract
return JSON: { score: number, reason: string (max 15 words) }
no explanation. no preamble. just the JSON.
```

then in Clay I call haiku 4.5 via HTTP API, prefix the skill file contents to the user message, and pass the row data. the model reads the rubric, applies it, returns structured JSON. done.

total tokens per call: ~400-600 depending on skill length. haiku 4.5 costs roughly $0.0008 per 1000 input tokens. on a 10k row table that's about $5-8 for the full enrichment pass.

same task on opus 4.7 baseline (no skill, just a good system prompt): $90-140 for the same table. and I audited 200 rows across both. quality on structured rubric tasks: identical.

**where it actually breaks down**

haiku + skills doesn't win everywhere. it loses on:

- open-ended research (find me what this company actually does based on their site, not their About page copy)
- multi-signal synthesis (combine the job posting language + recent funding + tech stack signals into one ICP narrative)
- anything requiring inference from messy unstructured text

for those I use sonnet 4.6. still not opus. just sonnet + a looser skill that gives it room to reason.

opus stays in my stack for one thing: building the skills themselves. I prompt opus to write the scoring rubric, audit it, tighten the output contract. then haiku runs the rubric in production. that's the actual division of labor.

**the HubSpot side**

once the enrichment hits CRM I have a second skill layer for write operations. HubSpot property names are gnarly. instead of fumbling with API docs every time I have a skill called `hubspot-property-map.md` that lists every custom property I've created with its internal name, type, and accepted values.

the model reads that file, maps the enriched data to the right fields, and returns a PATCH payload. I pipe that into an HTTP API node in Clay and it writes directly to the company object. no manual field matching.

setup time: 2 hours to build the property map skill the first time. now every enrichment workflow just imports it. never touches HubSpot documentation again.

**tldr on what to actually do**

1. stop using agent teams for structured enrichment. one model, one call, one output contract.
2. build skill files for every repeating operation in your stack. rubric-based scoring especially.
3. run haiku 4.5 in production, sonnet for messier inference, opus only to build/audit the skills themselves.
4. map your HubSpot properties once in a skill file. stop letting models guess field names.

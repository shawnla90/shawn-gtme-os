ran a deterministic analysis across 12,356 public repos with CLAUDE.md files. the pattern is pretty brutal.

most of them read like this:

> "Be concise. Write clean code. Follow best practices. Prefer simple solutions."

Claude ignores all of it. not because it can't read, but because none of it constrains behavior. it's aspirational wallpaper. every instruction in that list is already baked into the base model. you're not changing anything.

---

**what actually works: executable constraints, not philosophy**

the instructions that change Claude's behavior are specific, testable, and hard to misinterpret. here's the difference in practice.

**vague (doesn't work):**
- "Write clean, maintainable code"
- "Be careful with destructive operations"
- "Follow the existing code style"

**executable (works):**
- "Never use `rm -rf` without prompting for confirmation first"
- "All new files go in src/. Never create files in root."
- "Use `pnpm` not `npm`. If you see a package-lock.json, stop and tell me."
- "Do not create new components. Edit existing ones."
- "Run `pnpm lint` after every file edit, before moving to the next step."

the second set triggers specific behavior at decision points. Claude has something to check against. the first set gives it nothing to check against.

---

**the three categories worth putting in CLAUDE.md**

after going through hundreds of repos that had measurably different output quality, the useful instructions fall into three buckets:

**1. file/directory rules**
where things live, what never gets touched, naming conventions. Claude makes a lot of file placement decisions automatically. if you don't constrain these, it'll invent structure.

```
- all database queries live in src/db/queries/. never inline SQL elsewhere.
- never edit files in /generated/. those are auto-generated on build.
- component files are PascalCase. utility files are kebab-case.
```

**2. tool/command rules**
what runs, what doesn't, what to do before and after. this is where most people leave the most performance on the table.

```
- use `gh` for all GitHub operations. never use the GitHub web URL approach.
- after editing any .ts file, run `pnpm typecheck` and fix errors before proceeding.
- never run `git push --force`. if you think you need to, stop and explain why.
```

**3. decision gates**
explicit points where Claude should stop and ask instead of proceeding. this one is underused.

```
- if you're about to delete more than 3 files, stop and list them first.
- if a task requires changing the database schema, stop and describe the migration needed before writing any code.
- if you're unsure which of two approaches fits the existing patterns, ask. don't pick one and proceed.
```

---

**what to cut from your CLAUDE.md right now**

- any sentence containing the word "best practices"
- anything that starts with "try to" or "prefer to"
- adjective-only instructions (clean, simple, maintainable, robust)
- anything that's a restatement of general coding principles

those aren't instructions. they're hopes.

for a while I was doing what everyone does. CLAUDE.md + a `/memory/` folder full of `.md` files. feedback memories, project context, user preferences. the whole thing.

it worked okay until it didn't.

the problems I kept hitting:

- context got stale. a memory file from 3 weeks ago had wrong info and the agent confidently used it
- retrieval was fuzzy. the model would "remember" things that weren't in any file, or miss things that were
- no diffs. I had no idea when a memory changed or what it said before
- file sprawl. 40+ .md files with overlapping content and no deduplication logic

so I rebuilt it on SQLite. local file, no server, no dependencies beyond the standard library. here's the actual setup.

---

**the schema**

```sql
CREATE TABLE memories (
 id INTEGER PRIMARY KEY AUTOINCREMENT,
 type TEXT NOT NULL CHECK(type IN ('user','feedback','project','reference')),
 name TEXT NOT NULL,
 description TEXT NOT NULL,
 body TEXT NOT NULL,
 created_at TEXT DEFAULT (datetime('now')),
 updated_at TEXT DEFAULT (datetime('now')),
 superseded_by INTEGER REFERENCES memories(id)
);

CREATE TABLE memory_diffs (
 id INTEGER PRIMARY KEY AUTOINCREMENT,
 memory_id INTEGER REFERENCES memories(id),
 old_body TEXT,
 new_body TEXT,
 changed_at TEXT DEFAULT (datetime('now'))
);

CREATE VIRTUAL TABLE memories_fts USING fts5(
 name, description, body,
 content=memories,
 content_rowid=id
);
```

three tables. `memories` is the source of truth. `memory_diffs` is the audit trail. `memories_fts` is a full-text search index so retrieval isn't just vibes.

```python
import sqlite3
import json

def store_memory(db_path, key, value, agent_id):
 conn = sqlite3.connect(db_path)
 cur = conn.cursor()
 cur.execute("INSERT OR REPLACE INTO memories (key, value, agent_id) VALUES (?, ?, ?)",
 (key, json.dumps(value), agent_id))
 conn.commit()
 conn.close()
```

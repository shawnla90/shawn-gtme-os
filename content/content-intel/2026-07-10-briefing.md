### REDDIT Version

stop treating context window full as a reset. here's the handoff system I actually use

most people hit the context limit, see the warning, and either (a) start a fresh chat and re-explain everything from scratch or (b) keep cramming tokens in until Claude goes sideways on them.

both are painful. neither is necessary.

the fix is a structured handoff file. one document that captures the session state so the next window picks up clean.

here's the format I've landed on after a few months of running long Claude Code sessions:

---

**HANDOFF.md**

```
## session state
date: [today]
task: [one line . what are we actually doing]
status: [in progress / blocked / ready for review]

## what's done
- [specific thing completed, with file path if relevant]
- [another thing]

## decisions made
- chose X over Y because [reason] . don't revisit this
- [other decision + why]

## current context
[paste the 3-5 most relevant code blocks or schema snippets the next session needs to see immediately]

## next steps (in order)
1. [specific next action]
2. [then this]
3. [then this]

## known blockers / watch-outs
- [thing that's going to bite the next session if it doesn't know]
```

---

when Claude warns me context is getting full, I tell it:

> "write a HANDOFF.md using this format. capture everything the next session needs to continue without losing ground."

then new chat, first message:

> "read HANDOFF.md and continue the task."

that's it. new session reads the file, has the decisions, has the next steps, picks up where we left off.

**the decisions made section is the most important part.** that's where all the "we already tried that and it broke X" lives. without it, a fresh context will confidently suggest the exact thing you spent 45 minutes ruling out.

---

a few patterns I've noticed:

if the task is long (multi-hour build), I'll generate HANDOFF.md every 30-40 minutes as a checkpoint, not just at context limit. means I can kill the session anytime and not lose the thread.

for complex projects I keep a `/memory` folder and let the handoff file accumulate. HANDOFF.md always reflects current state. older ones archive as `HANDOFF_2025-07-09.md`. Claude Code can scan the folder when it needs history.

if you're using claude.md already, the handoff file is a complement not a replacement. claude.md is the static project context (architecture, conventions, how we work). HANDOFF.md is the dynamic session state (where we are right now, what happened today).

---

the real unlock here is treating your AI sessions the same way a good dev treats a git commit: leave a message. future you (and future context) will thank you.

3. find any section that's doing too many things at once (scope creep)

what it surfaced on mine:

**conflict**: I had "always ask before making changes to core files" and "don't interrupt the workflow with clarifying questions unless strictly necessary" in two different sections. **vague instruction**: "use good judgment on file structure." this is worthless. Claude said (paraphrasing) "this gives me no signal and forces me to invent a preference you haven't expressed. I'll get it wrong at least some of the time." I replaced it with an actual file tree pattern.

**scope creep**: I had a section called "code quality" that had grown into 14 bullet points covering style, testing, comments, error handling, commit messages, and PR descriptions. **noise**: I had three sentences explaining *why* I wanted a certain behavior. context that felt useful when I wrote it but Claude doesn't need the why . it needs the what. those sentences were adding tokens and sometimes getting treated as additional constraints.

the fix was mostly just editing. split the scope-creep section into four focused ones. deleted the explanatory prose. made the vague instructions specific. resolved the two conflicts by just deciding what I actually wanted.

the insight that stuck: Claude can't tell you when it's confused by your instructions unless you ask. it just... handles it, usually by averaging across the conflicting signals. that averaging looks like "slightly off judgment" from your side and you end up blaming the model.

the bug exposed something most builders were already doing wrong. context bleed isn't just a bug problem. it's a workflow problem.

here's what I run now.

**the four things that actually eat your context**

1. fat CLAUDE.md files with everything dumped in at session start
2. long-running sessions you never /compact or end
3. tool results that return walls of JSON you don't need
4. asking claude to hold state it shouldn't be holding at all

**what I changed**

**CLAUDE.md split by scope**

I have three now: a root CLAUDE.md that's under 200 lines (project overview, stack, anti-patterns, critical rules), a `/skills/CLAUDE.md` for skill-level context, and per-feature CLAUDEs I drop in temporarily when I'm heads-down on something specific. the root file loads every session. everything else loads only when relevant.

kept trimming the root until it was only things claude genuinely needs at session start vs. things it can figure out from reading code.

**task scoping before I type anything**

before I start a session I write down what I actually need done in that session. not the whole feature. not the refactor I've been putting off. the specific task. this sounds obvious. it changes everything about how you prompt and how fast context fills.

**subagents for research**

anytime I need claude to explore code, search for a pattern, or read a bunch of files to answer a question, I use the Agent tool with subagent_type=Explore. that work happens in a separate context. the result comes back clean. main context stays lean.

**TodoWrite as external memory**

instead of re-explaining what's done and what's next every few prompts, I use TodoWrite at the start of sessions and update it as I go. claude tracks state in the todo list, not in conversation history. this alone cut my average session length significantly.

**aggressive /compact usage**

I /compact before switching focus areas within a session. not just when the context bar turns red. early and often. the summary it generates is usually better context than the raw conversation anyway.

**the check I run when sessions feel slow**

when responses start feeling slightly off or claude seems to be losing track of something simple, that's usually the signal. I'll ask it directly: "how full is our context right now?" and it'll give an honest read. if we're over 60-70%, I /compact or end the session and restart with a clean brief.

**one thing I stopped doing**

pasting full file contents into the prompt when I could just give claude the path and let it read. the Read tool exists. use it. same with search results, API responses, anything long. pass references, not contents, wherever possible.

none of this is revolutionary. it's just the boring session hygiene that keeps production builds from going sideways. the bug made it obvious. the workflow is what keeps it from happening again.

**why sequential wins in practice:**

the tasks where parallel actually helps: pure data retrieval with no downstream synthesis. spinning up 5 agents to scrape 5 URLs simultaneously and dump raw results. totally valid. but that's not orchestration, that's just async HTTP calls with extra steps.

**what I'm running now:**

single orchestrator. sequential subagents. explicit context passing between steps. each subagent gets the full output of the previous one as part of its system prompt.

here's what I built after my incident:

**layer 1: budget alerts aren't guardrails**

google cloud budget alerts will email you after you've already spent the money. same with most billing dashboards. they're forensic, not preventive. I see people treat alerts as safety nets and they're not . they're autopsies.

real guardrails are execution-time checks, not notification-time checks.

**layer 2: the $0 ceiling pattern**

for any agent that touches a paid API (openai, anthropic, gcp, apify, whatever), I now wrap the execution in a pre-flight spend check:

```python
def check_budget_before_run(service, daily_limit_usd):
 current_spend = get_today_spend(service) # pull from billing API
 if current_spend >= daily_limit_usd:
 raise BudgetExceeded(f"{service} at ${current_spend} . daily limit ${daily_limit_usd} hit")
 return True
```

this runs before any agent step that touches a paid resource. if the check fails, the workflow stops and logs it. no retry, no fallback, just stop.

**layer 3: token budgets at the model level**

anthropic's API has `max_tokens` but people don't use it like a budget . they use it like a safety valve. I now set it explicitly tight on any agent that's looping:

```python
client.messages.create(
 model="claude-opus-4-6",
 max_tokens=800, # not 4096 "just in case"
 ...
)
```

if the task needs more than 800 tokens to answer, it's probably doing too much in one step anyway. break it up.

**layer 4: loop exit conditions that aren't "until done"**

the $25k bill story had an agent running indefinitely. the exit condition was "until the task is complete" and the task never registered as complete.

every loop I write now has an explicit iteration ceiling:

```python
MAX_ITERATIONS = 50
for i in range(MAX_ITERATIONS):
 result = agent_step()
 if result.done:
 break
 if i == MAX_ITERATIONS - 1:
 log_incomplete(result) # flag it, don't silently fail
```

50 is still a lot. but it's not infinite.

**layer 5: separate IAM for agents**

this one's more setup cost but worth it. agent credentials get their own service account with billing quotas attached at the account level. even if the code guardrails fail, the account-level quota will hard stop it. defense in depth.

## 1. [TUTORIAL] I ripped out all my MCPs and replaced them with CLIs. here's the exact stack

**Thesis:** MCPs introduce friction, parameter errors, and context bloat that CLIs solve in one line. After switching, Claude Code sessions got tighter and the token burn dropped noticeably.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=8 voice=9 gap=6 effort=medium

### REDDIT Version

been running 17 MCP servers for a few months. then I started actually measuring context usage at session start.

every MCP loads its full tool definitions before Claude writes a single line. the filesystem MCP alone dumps ~40 tools into schema space. multiply that by 17 servers and you're burning tokens on tools that might never get called in that session. then add parameter validation errors when Claude calls something like `mcp__github__create_pull_request` with a slightly wrong shape and you lose 3-4 turns just recovering.

CLIs don't have that problem. `gh pr create` either works or it tells you why in plain text.

here's the swap list:

**GitHub MCP → gh CLI**

Claude already knows gh syntax cold. zero parameter errors in the two weeks since I switched. the MCP was giving me 38 tool definitions in context to do what `gh pr create --fill` does in one line.

**filesystem MCP → nothing**

Claude Code has native file access. the filesystem MCP exists to grant access Claude Code already has. this one felt obvious in hindsight.

**HubSpot MCP → curl wrapper**

I keep a small helper at monorepo root:

```bash
#!/bin/bash
# hs.sh
curl -s -H "Authorization: Bearer $HUBSPOT_API_KEY" \
 -H "Content-Type: application/json" \
 "$@"
```

CLAUDE.md gets one line: "use ./hs.sh for HubSpot API calls." no MCP initialization, no tool schema bloat.

**Postgres MCP → psql**

.env has:
```
DB_URL=postgres://user:pass@host/dbname
```

CLAUDE.md: "use `psql $DB_URL -c` for database queries." done.

**Brave Search MCP → built-in WebSearch**

the WebSearch tool that ships with Claude Code covers 90% of what I was using the Brave MCP for. the other 10% is a direct curl to the API.

**Apify MCP → Apify CLI**

```bash
apify call username/actor-name --input='{"url":"..."}'
```

Claude knows the syntax. it doesn't need a schema to tell it how.

what I still run as MCPs: sequential-thinking (no CLI equivalent, actually useful for planning mode) and memory (the tagging layer adds something the raw files don't). everything else is gone.

the key insight is that CLAUDE.md is doing the heavy lifting now instead of MCP schemas. a short section listing which CLI handles which system, and Claude figures out the syntax. it already knows gh, psql, curl, apify. it doesn't need an MCP to describe the interface.

token difference: complex builds were running 180-220k at `/cost`. same session types now land at 120-150k. some of that is other changes but the MCP culling made a visible dent.

worth auditing your own list. anything that's basically a wrapper around a CLI or REST API, test whether Claude can call the underlying thing directly. most of the time it can, and the session is cleaner for it.

## 2. [TUTORIAL] stop writing massive CLAUDE.md files. use compaction boundaries instead

**Thesis:** Context drift on day 2 of a project isn't a CLAUDE.md problem, it's a session architecture problem. Compaction boundaries fix it permanently without the maintenance overhead.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=7 voice=8 gap=7 effort=medium

### REDDIT Version

stop writing massive CLAUDE.md files. use compaction boundaries instead

spent last week auditing why my day 2 sessions kept drifting. the claude.md was 600 lines. still losing context on complex builds. added more to claude.md. still drifting.

turns out the problem wasn't the file. it was where i was putting things.

**the actual problem**

CLAUDE.md is loaded at session start. that's it. everything that happens during the session -- decisions made, approaches tried, things explicitly ruled out -- lives only in conversation history. when compaction kicks in and compresses that history, those decisions get lossy. sometimes they disappear entirely.

so day 2 you re-explain the same architectural choice you made on day 1. claude suggests the thing you already ruled out. the drift isn't forgetting project structure. it's forgetting session decisions.

cramming more into CLAUDE.md doesn't fix that. it's solving the wrong problem.

**what compaction boundaries actually are**

a compaction boundary is a deliberate checkpoint written to a file at a natural session transition. not a summary. a living document that reflects current decisions, not history.

the pattern i use:

```
.claude/
 CLAUDE.md # static: architecture, conventions, commands, tool stack
 decisions.md # dynamic: active decisions + what was ruled out + why
 session-state.md # volatile: what's in progress right now
```

CLAUDE.md stays short. maybe 80 lines. it points to the other files:

```markdown
## Active Context
- Current decisions: .claude/decisions.md
- In-progress work: .claude/session-state.md
```

**how it works in practice**

at the end of any session where a real decision was made, i ask claude to update decisions.md:

```
update .claude/decisions.md with:
- we're using approach X for the auth flow
- approach Y was ruled out because of Z
- next session starts at [specific file/function]
```

that file survives compaction because it's a file, not conversation history. next session, claude reads it. the decision is there. you don't re-litigate.

session-state.md is even simpler. just current work in progress. i wipe it at the start of each session and rebuild it as context develops. takes 30 seconds.

**the claude.md you actually need**

after moving decision state into its own file, my CLAUDE.md went from 600 lines to 74. it now has:

- project structure (where things live)
- tech stack and versions
- commands to run tests/build/dev
- critical conventions (naming, patterns)
- pointer to decisions.md

that's it. anything about current state lives in the dynamic files.

**why this holds up through compaction**

when claude compacts the session history, it summarizes the conversation. but it doesn't delete your files. so if the decisions are written to disk before compaction hits, they're there when the next read happens.

the boundary is the moment you write the file. everything before that moment can compress. the file is the anchor point.

**one more thing**

if you're on a long session and feel drift starting, don't add to claude.md. instead:

```
before we continue -- write a current-state snapshot to .claude/session-state.md
include: what we're building, what's decided, what's next, what's explicitly off the table
```

then continue. that snapshot acts as a mid-session compaction boundary. works better than re-prompting and way better than hoping the summary kept the right details.

the maintenance overhead is nearly zero because decisions.md only grows when you actually make decisions. it's not documentation for its own sake. it's a record of choices that survived the build.

try it on your next complex session and see if day 2 feels different.

## 4. [HOT_TAKE] DESIGN.md is the CLAUDE.md nobody's talking about

**Thesis:** Every AI builder obsesses over CLAUDE.md for behavior consistency. nobody has a DESIGN.md for visual consistency, and that's why AI-built UIs all look identical after 3 iterations.
**Target sub:** r/r/vibecoding
**Scores:** viral=7 voice=8 gap=8 effort=low

### REDDIT Version

every AI-built UI starts looking the same after 3 iterations.

not because the model can't design. because you never told it what your design system is.

you've probably spent real time tuning your CLAUDE.md. behavior instructions, tool permissions, response formats, coding conventions. and it works. Claude stays consistent.

but then you ask it to build a new component and suddenly your spacing is off, the button radius doesn't match, the font weight is wrong, and you've got two slightly different shades of the same gray fighting each other.

that's not a model problem. that's a missing file problem.

DESIGN.md is the equivalent file for your visual system.

here's what mine has:

- color tokens (not hex codes, the token names and their usage context: primary action vs. destructive vs. muted text)
- spacing scale (I use a 4px base, but I spell out what "small gap", "section padding", "page margin" actually means in px so it's not interpreted)
- component patterns (how a card is structured, when to use a sheet vs. a dialog, what belongs in a sidebar)
- typography rules (which weight is body copy, which is label, what the h1 treatment looks like)
- what we never do (gradients on text, drop shadows on flat components, centered body copy)

when Claude has this file in context, the 5th component looks like the 1st. without it, by iteration 3 you're firefighting visual drift instead of shipping features.

took me 45 minutes to write the first version. I grabbed a component I was happy with, asked Claude to extract the design rules from it, then edited for accuracy. most of it came out right.

the "what we never do" section is underrated. constraints are as useful as instructions. maybe more.

if you're vibe-coding UIs and wondering why everything feels frankensteined after a few sessions, check whether you've given your model a design language to work from.

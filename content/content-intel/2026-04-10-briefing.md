three stacked bugs have been silently killing extended thinking in Claude Code since v2.0.64, and most people have no idea it's not working.

I spent two days convinced my prompts were just bad. turns out the feature was never actually engaging. here's how to verify and fix it.

**how to tell it's broken**

run this in your terminal while Claude Code is active:

```bash
tail -f ~/.claude/logs/claude-code-*.log | grep -E "thinking|budget|extended"
```

if extended thinking is actually engaging, you'll see something like:

```
[thinking] budget_tokens: 10000, used: 8432
```

if you see nothing, or you see:

```
[thinking] budget_tokens: 0
```

it's broken. the model is running in standard mode regardless of what you set.

**the three bugs**

**bug 1: the model override silently drops thinking params**

if you have `model` set anywhere in your `.claude/settings.json`, the thinking configuration doesn't get passed through correctly. the model override handler strips the extended thinking params before the API call goes out.

**bug 2: `maxThinkingTokens` isn't the right key post v2.0.64**

the config key changed. most guides (including Claude's own docs at the time of that release) still reference `maxThinkingTokens`. the correct key is now `thinkingBudget`. using the old key fails silently -- no error, just no thinking.

**bug 3: the effort level flag and the budget conflict**

if you set `"effort": "max"` AND manually set `thinkingBudget`, they fight each other. the effort flag sets its own budget internally, and then sees a manual override it doesn't know how to reconcile. result: it zeroes out and falls back to standard.

**the fix**

working `.claude/settings.json` config:

```json
{
 "model": "claude-opus-4-5",
 "thinking": {
 "type": "enabled",
 "budget_tokens": 10000
 }
}
```

don't use `effort: max`. don't use `maxThinkingTokens`. don't set model and thinking budget in separate config files that get merged -- pick one place.

after you update, restart Claude Code fully (not just a new session, actually kill the process) and re-run the log grep above to confirm the budget tokens are showing up.

**how I found this**

was building a complex Clay-to-HubSpot scoring table and kept getting shallow outputs from what I thought was max thinking mode. started logging API calls to compare token usage between sessions where I manually prompted for deep analysis vs. ones where I expected the config to handle it.

standard mode: 2-3k output tokens. what I thought was extended thinking mode: also 2-3k. should've been 10k+.

that's when I realized the config was never taking effect.

confirmed it with the log grep above, then started bisecting the config keys one at a time until the behavior changed.

### REDDIT Version

how I use Claude Code hooks to cut token burn by ~80% on large codebases

this took me embarrassingly long to figure out. I was watching my context window evaporate on every session in my monorepo (50+ skills, 17 MCP servers, deeply nested structure) and kept wondering why.

turned out Claude was grep-crawling the entire codebase to find function definitions, imports, type references. every navigation op was burning hundreds of tokens on file content it didn't need.

the fix: a PreToolUse hook that intercepts Bash calls matching grep/find patterns and forces a redirect to LSP instead.

here's the actual setup.

**1. the hook config in `.claude/settings.json`**

```json
{
 "hooks": {
 "PreToolUse": [
 {
 "matcher": "Bash",
 "hooks": [
 {
 "type": "command",
 "command": "node ~/.claude/hooks/lsp-guard.js"
 }
 ]
 }
 ]
 }
}
```

**2. `lsp-guard.js`**

```js
const input = JSON.parse(require('fs').readFileSync('/dev/stdin', 'utf8'));
const cmd = input?.tool_input?.command || '';

const grepPatterns = [
 /\bgrep\b.*-r/,
 /\brg\b/,
 /\bfind\b.*-name.*\.(ts|js|py)/,
 /grep.*function\b/,
 /grep.*import\b/,
 /grep.*class\b/,
];

const shouldIntercept = grepPatterns.some(p => p.test(cmd));

if (shouldIntercept) {
 console.log(JSON.stringify({
 action: "block",
 reason: `use LSP tools instead of grep for code navigation. 
- finding a definition? → LSP GoToDefinition
- finding references? → LSP FindReferences 
- searching symbols? → LSP WorkspaceSymbols
grep is for log files and data. not source navigation.`
 }));
} else {
 console.log(JSON.stringify({ action: "allow" }));
}
```

when Claude tries to run `grep -r "handleAuth" src/`, it gets blocked with the redirect instruction. it pivots to LSP, finds the symbol in one shot, reads only the file it needs.

**3. why this hits so hard in monorepos**

in a flat project, grep isn't terrible. in a nested monorepo with shared packages and deep imports, a single `grep -r` on a type name can touch 200+ files before surfacing anything useful. LSP resolves the same lookup with zero file reads.

the ~80% number is real but context-dependent. for sessions heavy on refactoring or cross-package tracing, it's closer to that. for sessions that are mostly writing new code, less dramatic.

**4. the monorepo-specific addition**

I added a second layer that fires on any find command targeting the whole repo root, since I have a bad habit of doing `find . -name "*.ts"` when I should just use Glob:

```js
const findAllPatterns = [
 /^find \. -name/,
 /^find \/ -name/,
];

if (findAllPatterns.some(p => p.test(cmd))) {
 console.log(JSON.stringify({
 action: "block", 
 reason: "use the Glob tool for file pattern matching, not find. faster and token-safe."
 }));
}
```

**5. what I learned building this**

the hook action: "block" + reason pattern is underused. most examples I saw use hooks for logging or running side effects. but using them as guardrails that redirect Claude's tool choice is way more powerful for token efficiency.

Claude takes the reason text seriously. if you're specific about what to use instead, it pivots cleanly without losing context on the task.

## 7. [PSA] don't trust AI to test AI. learned this the hard way.

**Thesis:** If Claude writes the code and Claude grades the output, you don't have a test — you have a mirror. Real validation means deterministic checks, not an LLM saying the math looks right.
**Target sub:** r/r/ClaudeAI
**Scores:** viral=7 voice=8 gap=8 effort=low

### REDDIT Version

don't trust AI to test AI. learned this the hard way.

built a scoring pipeline last month. Claude enriches a lead, Claude scores it 1-10 based on fit criteria, Claude writes a summary explaining the score. felt solid. shipped it.

then I ran 200 records through it and eyeballed the outputs. everything looked... fine? coherent? reasonable?

took me three days to realize I had no idea if it was actually working correctly. I was asking Claude to check Claude's work. of course it looked right. it was explaining itself to itself.

the real problem: there's no ground truth. if the scoring logic drifts because of a prompt tweak or a model update, nothing breaks loudly. it just silently gets worse and you won't catch it until a client notices the bad leads.

what actually fixed it:

**deterministic assertions on known inputs.** I built a small labeled dataset of 30 leads I personally scored. ICP = 1, not ICP = 0. run the pipeline, compare. if accuracy drops below 80%, the pipeline doesn't ship.

**unit test the extractors, not the summaries.** if I'm pulling "number of employees" from a webpage, the test checks that the output is an integer in a reasonable range. not "does this look like a reasonable employee count to you, Claude."

**log the raw model outputs, not just the final answers.** when something's off, I need to see what the model actually returned, not what my parsing layer turned it into.

**canary records.** a handful of records with known expected outputs that run every time. if the canary breaks, something upstream changed.

the summary being "well-written" is not a test. Claude will always write a coherent explanation for whatever answer it gives, even if the answer is wrong.

if you're building eval pipelines, scoring agents, or anything where an LLM grades another LLM's work, you need at least one layer of ground truth that doesn't involve asking an AI if the AI did a good job.

otherwise you just have a mirror that sounds confident.

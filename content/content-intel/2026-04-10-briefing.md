here's how it actually works and how I'm using it in Clay and HubSpot workflows.

**the core mechanic**

Opus 4.6 is expensive but thinks in ways Sonnet doesn't. Sonnet 4.6 is fast, cheap, and executes reliably. the advisor pattern splits the job: Opus sees the problem and produces a structured plan, Sonnet follows that plan to completion.

in Claude Code, you can set this up in CLAUDE.md:

```
advisor model: claude-opus-4-6
executor model: claude-sonnet-4-6
```

or call it directly via the SDK:

```python
# step 1: get the plan from Opus
advisor_response = client.messages.create(
 model="claude-opus-4-6",
 max_tokens=1024,
 system="you are a GTM workflow architect. produce a numbered execution plan only. no prose.",
 messages=[{"role": "user", "content": task}]
)

plan = advisor_response.content[0].text

# step 2: execute with Sonnet using the plan as context
executor_response = client.messages.create(
 model="claude-sonnet-4-6",
 max_tokens=4096,
 system="execute the following plan exactly. return structured output only.",
 messages=[{"role": "user", "content": f"plan:\n{plan}\n\ntask:\n{task}"}]
)
```

the key is the system prompt split. Opus gets told to think, not do. Sonnet gets told to do, not think.

**how I'm using this in Clay**

I have a Clay table that pulls companies from a LinkedIn scrape, runs them through a Claygent, and writes personalized outreach. the old setup was a single HTTP API call to Opus for the whole thing. $0.15+ per row at scale.

new setup:

- HTTP API call 1 (Opus): takes the company URL, LinkedIn headline, and recent posts. returns a 5-point account analysis: ICP fit score, buying signals, pain point hypothesis, personas to target, outreach angle. ~300 tokens out.
- HTTP API call 2 (Sonnet): takes that analysis as context + the rep's name + product positioning. writes the actual email sequence. ~800 tokens out.

cost per row dropped from ~$0.15 to ~$0.04. output quality is basically the same because Sonnet isn't guessing the strategy anymore. it's executing a pre-built one.

the Opus prompt I use:

```
you are a B2B account researcher. analyze the following company data and return JSON only:

{
 "icp_score": 1-10,
 "buying_signals": ["signal1", "signal2"],
 "pain_hypothesis": "one sentence",
 "personas": ["title1", "title2"],
 "outreach_angle": "one sentence"
}

company data:
{{company_data}}
```

then in the Sonnet call I pass the whole JSON as `{{opus_output}}` and let it write from there.

**HubSpot side**

I run a similar pattern for HubSpot enrichment workflows. when a new company enters a lifecycle stage in HubSpot (say, MQL), a webhook fires to a small FastAPI endpoint. that endpoint:

1. pulls the company properties from HubSpot (industry, headcount, tech stack from Clearbit, recent activity)
2. sends to Opus for account strategy output
3. sends Opus output + raw data to Sonnet for property writes
4. pushes structured fields back to HubSpot: `gtm_angle`, `priority_persona`, `outreach_sequence_type`

the reps see pre-filled context when they open the record. they're not starting from scratch.

this runs on ~2,400 MQL enrichments a month. before: $360/month in Claude API costs using Opus for everything. after: ~$80. same output fidelity because the strategy layer (Opus) is actually the small part. writing at volume is Sonnet's job.

**the pattern that matters**

thinking is expensive. execution is cheap. most people run everything through their most capable model because it feels safer. but if you can cleanly separate what requires judgment from what requires throughput, you stop paying Opus rates for Sonnet-level tasks.

background: I run a content pipeline that watches HubSpot for lifecycle stage changes, checks Apify scrape job completion, and picks up Instantly campaign replies. all three were burning tokens on sleep → check → repeat cycles every 30 seconds. just sitting there spinning.

the Monitor tool fixes this. instead of the agent running a loop, you spin up a background watcher that only wakes the agent when a condition actually becomes true. real sleep between events. no idle token burn.

here's the Apify job watcher I set up first:

```json
{
 "monitor": {
 "type": "http_poll",
 "url": "https://api.apify.com/v2/actor-runs/{{runId}}",
 "headers": { "Authorization": "Bearer {{token}}" },
 "condition": "$.status === 'SUCCEEDED'",
 "interval_ms": 15000,
 "timeout_ms": 300000
 }
}
```

when the condition hits, the monitor wakes the agent and passes the full response payload. everything in between is actual silence.

for the HubSpot lifecycle watcher I switched to the webhook variant instead of polling:

```json
{
 "monitor": {
 "type": "webhook",
 "path": "/hooks/hs-lifecycle",
 "filter": "$.properties.lifecyclestage === 'opportunity'"
 }
}
```

point your HubSpot workflow action at a local tunnel (I use cloudflared) and the agent only fires when the filter condition matches. everything else drops at the edge before it ever reaches the agent.

three things that bit me setting these up:

**interval_ms compounds fast.** I started at 5000ms on Apify and it was fine in isolation. with 8 concurrent runs it was hitting rate limits. 15000ms is the production sweet spot for that API.

**condition syntax is JSONPath.** if you've written Clay formulas or jq this is fast to pick up. if you haven't, the first trip-up is writing `data.status` instead of `$.data[0].status`. took me 20 minutes to figure out why a valid condition wasn't matching.

**set timeout_ms explicitly.** the default is lower than you'd expect. if your job runs longer than the timeout the monitor silently dies and the agent never wakes. I use 300000 (5 min) baseline and bump it for scrapes that run long.

actual token delta before/after: the three polling loops were running ~2,400 tokens/hour just sitting idle. post-migration: ~120/hour. not a huge cost thing on its own but across a pipeline that runs 16 hours a day it adds up, and the bigger win is the agent isn't stuck in a loop when something downstream actually needs it.

### REDDIT Version

every week there's a new thread: "opus degraded again," "claude got dumb overnight," "switching to gemini."

I get it. I've been there. you build a workflow that works beautifully for two weeks, then something shifts and suddenly your prompts return garbage and you're back to square one.

but the actual problem isn't the model. it's that most people are building on a single-point-of-failure architecture and treating a non-deterministic system like it's a database.

here's what a more resilient setup looks like in practice:

**version pin your model calls**

stop using `claude-opus-latest` in production workflows. pin to a specific model ID. `claude-opus-4-5-20251001` or whatever version was working. yes it'll drift eventually but you control when.

```python
model = "claude-opus-4-5-20251001" # pinned, not latest
```

**add a fallback chain**

if your primary model errors or returns a confidence score below threshold, route to your fallback. sonnet is absurdly capable for 90% of tasks anyway.

```python
def run_with_fallback(prompt, primary="claude-opus-4-6", fallback="claude-sonnet-4-6"):
 try:
 result = call_model(primary, prompt)
 if not passes_quality_check(result):
 return call_model(fallback, prompt)
 return result
 except Exception:
 return call_model(fallback, prompt)
```

**build quality checks into the pipeline, not at the end**

don't just return the LLM output. validate it. if you're extracting structured data, check that required fields exist. if you're generating copy, run a quick regex check on format. catch degradation at the call level, not when a client complains.

**log model + version on every call**

you can't debug "claude got dumb" if you don't know which model version ran on which call. one extra field in your logs saves hours of debugging.

```python
log.info(f"model={model} prompt_tokens={usage.input_tokens} output_tokens={usage.output_tokens}")
```

the teams I see thriving aren't the ones who found a perfect model. they're the ones who built systems that degrade gracefully instead of failing catastrophically.

model variance is a constant. build for it.

### REDDIT Version

Claude Code forgets everything between sessions. here's the memory system I built so it doesn't.

CLAUDE.md is the default answer everyone gives. put your context there, done. but CLAUDE.md is static. it's great for style rules and repo conventions. it's useless for capturing what actually happened . the decisions you made, the approaches that failed, the project state as of last Thursday.

so I built a file-based memory system. nothing fancy. just a structured directory that Claude reads and writes to across sessions.

here's how it works:

**the structure**

```
~/.claude/projects/your-project/memory/
├── MEMORY.md ← index only, always loaded
├── user_context.md
├── feedback_no_mocks.md
├── project_auth_rewrite.md
└── reference_linear_board.md
```

MEMORY.md is just an index. pointers to files with one-line descriptions. it's always in context. the actual content lives in typed files.

**the types**

four memory types, each serving a different purpose:

`user` . who's working here, what they already know, how they prefer to collaborate. if you're a senior Go engineer who's new to React, Claude should explain frontend things differently. this captures that.

`feedback` . corrections you've given. the format matters: lead with the rule, then a **Why:** line, then a **How to apply:** line. "don't mock the database in tests" is useless without "because a mocked test passed last quarter while the real migration silently broke." the why is what makes it apply to edge cases instead of just repeating the exact scenario.

`project` . ongoing decisions, motivations, constraints. not stuff derivable from code or git history. things like "auth middleware rewrite is driven by legal compliance, not tech debt" changes every suggestion Claude makes about scope. always convert relative dates to absolute when saving . "Thursday" becomes "2026-04-10" so it's still readable in 3 months.

`reference` . external pointers. which Linear board, which Grafana dashboard, which Slack channel. so when you say "check the pipeline bugs" Claude knows where to look.

**what NOT to save**

this is where most people overcomplicate it. don't save:
- code patterns or architecture (just read the code)
- git history or who changed what (git blame exists)
- debugging solutions (the fix is in the code, the commit message has context)
- anything in CLAUDE.md already
- in-progress task state (that's what todos are for, not memory)

memory is for things that aren't derivable from the current state of the repo.

**how Claude writes memories**

I added instructions to my global CLAUDE.md that tell Claude when to save each type. the key trigger for feedback memories is anything that looks like a correction . "no, don't do that", "let's not...", "stop doing X". Claude writes the file and updates the index in the same step.

the frontmatter format for each file:

```markdown
---
name: feedback_no_database_mocks
description: integration tests must hit real db . mock/prod divergence burned us last quarter
type: feedback
---

don't mock the database in integration tests.

**Why:** prior incident where mocked tests passed but the real migration failed silently in prod.

**How to apply:** any test touching data access . use a real test database, not mocks.
```

**how Claude reads memories**

MEMORY.md stays in context at all times (keep it under 200 lines or it gets truncated). the actual memory files only get read when they seem relevant to the task. Claude checks the index, pulls what applies, ignores the rest.

the result is sessions that feel continuous. Claude knows you pushed back on a pattern two weeks ago and why. it knows the project constraint driving the current refactor. it knows which external systems to reference instead of asking you again.

CLAUDE.md handles the static stuff. this handles everything else.

I've logged 6,852 Claude Code sessions. tracked output length, task completion rate, refusal frequency, code correctness on a fixed test suite, and hallucination rate on known-answer prompts. degradation is real. I can show you the charts.

but that's not what I see in these posts.

what I see is "it feels worse" and "it used to just do this" and "something changed last Tuesday." that's not measurement. that's vibes. and the problem with vibes is they compound. a bad session makes you more suspicious in the next one, which changes how you prompt, which changes the output, which confirms the vibe.

here's the actual pattern I've noticed in the 6,852 sessions: the workflows that break on model updates were fragile before the update. they were brittle to begin with. they only worked because one specific model version had one specific behavior that you accidentally depended on.

if Claude updating its refusal thresholds breaks your entire pipeline, you built a pipeline that depends on a specific refusal threshold. that's on the architecture, not the model.

the stuff in my stack that survives every model change:

- explicit context in CLAUDE.md that doesn't assume behavior
- structured outputs with validation, not "just return JSON and hope"
- task decomposition where each step has a clear success condition I can actually check
- skills and workflows with retry logic that doesn't assume one-shot perfection

the stuff that breaks: one-shot "do this whole thing" prompts with no guardrails that happened to work for three months because the model was lenient on a particular edge case.

I'm not saying degradation isn't happening. it is. I have the data. specific capabilities have gotten worse. context handling at long windows, consistency on multi-step instructions, some coding tasks.

but if every model release breaks your day, the model is a dependency you haven't hardened against. you'd never build a production system that assumes a third-party API never changes behavior. why would you assume that about a model?

measure the real things. build to the interface, not to the quirk.

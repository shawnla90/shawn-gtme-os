here's my exact setup in `~/.claude/settings.json`:

```json
{
 "hooks": {
 "PreToolUse": [
 {
 "matcher": "Bash",
 "hooks": [
 {
 "type": "command",
 "command": "echo \"$CLAUDE_TOOL_INPUT\" | jq -r '.command' | grep -qE '(rm -rf|git push --force|DROP TABLE)' && echo 'BLOCKED: destructive command detected' && exit 1 || exit 0"
 }
 ]
 }
 ],
 "PostToolUse": [
 {
 "matcher": "Bash",
 "hooks": [
 {
 "type": "command",
 "command": "if [ \"$CLAUDE_TOOL_RESPONSE_EXIT_CODE\" != \"0\" ]; then echo \"command failed with exit $CLAUDE_TOOL_RESPONSE_EXIT_CODE\" >> /tmp/claude-errors.log; fi"
 }
 ]
 }
 ]
 }
}
```

what this actually does:

**PreToolUse on Bash** . blocks specific destructive patterns before they execute. claude never has to ask "are you sure?" because it just gets a BLOCKED response with context. it reads that, adjusts, keeps moving.

**PostToolUse on Bash** . logs every non-zero exit to a file. this is underrated. when a 3-hour run finishes, I can open `/tmp/claude-errors.log` and see exactly where things got weird without reading the full transcript.

the second piece I added more recently . a Stop hook that pings me on Pushover when the run ends:

```json
"Stop": [
 {
 "hooks": [
 {
 "type": "command",
 "command": "curl -s --form-string 'token=YOUR_TOKEN' --form-string 'user=YOUR_USER' --form-string 'message=claude stopped: $(date)' https://api.pushover.net/1/messages.json"
 }
 ]
 }
]
```

now I start a run, walk away, get a notification when it's done. no babysitting.

the mental model shift: skip-permissions is about removing friction from claude. hooks are about giving claude better information so it doesn't need to stop in the first place.

a hook that returns exit 0 = proceed. exit 1 = stop with the message you printed. claude reads that output and course-corrects without escalating to you.

my longest uninterrupted run since setting this up was 4.5 hours building out a full data pipeline. zero interruptions. the PreToolUse hook blocked two commands that would've broken prod, logged them, and claude rerouted without me seeing any of it until the end.

## 10. [CONTRARIAN] the obsidian + claude code hype is missing the point. hook-based context injection is the actual move.

**Thesis:** People are excited about Obsidian as a knowledge base for Claude Code, but the real unlock is injecting domain-specific context via hooks based on the file path being touched — no manual notes, no vault maintenance, just the right context at the right time.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=7 voice=8 gap=8 effort=medium

### REDDIT Version

the obsidian + claude code hype is missing the point. hook-based context injection is the actual move.

every week I see another post about building a vault, linking notes, feeding your "second brain" to Claude Code. and sure, it's not wrong. but people are solving the wrong problem.

the real friction isn't that Claude doesn't know your domain knowledge. it's that it doesn't know *which* domain knowledge to apply *right now* based on what you're actually touching.

here's what I mean. if I'm editing a file in `/gtm-os/engine/`, I need Clay API context, Attio schema docs, and enrichment logic. if I'm in `/website/apps/shawnos/`, I need component patterns, routing conventions, and deployment constraints. if I'm in `/scripts/abm/`, I need the data pipeline rules and blocklist safety.

these are completely different contexts. an Obsidian vault doesn't solve this because you still have to pull the right notes at the right time . and you won't. you'll forget, or Claude will ignore the noise because the vault dump is too broad.

hooks solve this natively.

in `.claude/settings.json` you can define a `PreToolUse` hook that fires before Claude touches any file. write a shell script that checks the file path pattern and injects a context block. now Claude automatically gets the right context for the right folder, every time, without you maintaining a single note.

rough sketch:

```bash
#!/bin/bash
FILE_PATH="$1"

if [[ "$FILE_PATH" == *"/gtm-os/engine/"* ]]; then
 cat ~/.claude/contexts/clay-api-context.md
elif [[ "$FILE_PATH" == *"/scripts/abm/"* ]]; then
 cat ~/.claude/contexts/pipeline-safety-rules.md
fi
```

wire it up in settings.json:

```json
{
 "hooks": {
 "PreToolUse": [
 {
 "matcher": "Edit|Write",
 "hooks": [{ "type": "command", "command": "~/.claude/hooks/context-inject.sh $FILE_PATH" }]
 }
 ]
 }
}
```

now your context system is path-aware and automatic. no vault. no manual notes. no hoping Claude reads the right CLAUDE.md section.

the context files themselves are just markdown. 200-400 lines max. domain-specific rules, schema snapshots, API patterns, whatever that folder needs to work correctly.

the Obsidian crowd is building a knowledge retrieval system on top of a tool that already has a hook system. skip the middle layer.

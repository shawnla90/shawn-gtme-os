### REDDIT Version

most of my early Claude Code sessions burned 60-80k tokens before I'd shipped anything. the culprit wasn't the prompts. it was the session architecture.

after tracking usage across ~200 sessions, the pattern was clear: token waste clusters at the opening 10-15 exchanges, not during the actual build. here's what was happening and how I fixed it.

**the three main leaks**

**1. letting Claude explore instead of front-loading**

every time I opened a session with "I'm working on X, take a look at the codebase" . Claude would read CLAUDE.md, glob the directory, read 3-5 files to orient itself, ask clarifying questions, then read 2-3 more files based on answers.

that's 8-12 tool calls before anything happens. each Read on a 300-line file costs you.

the fix: open every session with a context block in your first message.

instead of:
```
I'm working on the dashboard app, take a look
```

write:
```
working in src/apps/dashboard/
key files: app/page.tsx (shell), lib/metrics.ts (data fetching), public/data/metrics.json (static source)
current state: metrics.json is stale . need a refresh cron that writes new data every hour
goal this session: build the cron script + wire the refresh endpoint
don't read anything yet, I'll tell you what we need
```

Claude skips the exploration entirely and starts planning against what you gave it.

**2. sequential file reads**

this one hurt me for months. I'd ask a question, Claude would read file A, answer, I'd ask a follow-up, it'd read file B. repeat.

each file read in a separate message = separate tool call overhead. Claude also re-confirms context between exchanges, which burns tokens silently.

batch them. if you know you're touching 5 files this session, say so upfront:

```
we'll need these files today:
- src/apps/dashboard/app/page.tsx
- src/apps/dashboard/lib/metrics.ts
- scripts/cron_runner.sh
- data/stats.json
read all of them now before we start
```

Claude fires parallel Read calls. single round trip. I've consistently saved 4-8k tokens per session just from this.

**3. mid-session CLAUDE.md re-reads**

CLAUDE.md loads automatically at session start. but if you say things like "check the project conventions" or "remind yourself of the workflow rules", Claude reads it again. mid-session re-reads add up fast on long instruction files.

trust that it was loaded. if you want Claude to apply a specific rule, quote it inline instead of sending it back to re-read the file. also watch for: "check if there's anything in the instructions about X" . that triggers a full re-read. just tell Claude the relevant instruction directly.

**the handoff pattern (biggest unlock)**

instead of cold-starting each session, I end every session by writing a handoff doc . what was built, what's in progress, which files are hot, what the next step is.

next session opens with:
```
read [handoff file path]
```

Claude gets full context without touching a single source file. session starts warm. exploration phase is basically zero.

the token difference between a cold start and a handoff-started session on the same project runs 15-25k tokens in my experience. on a $20/month plan that adds up fast.

**the practical checklist**

open every session with a context block (files, current state, goal, constraints). batch file reads upfront before the first real question. never ask Claude to re-read your instructions mid-session. use handoff docs to resume rather than re-explore.

none of this changes what you build. it just cuts the overhead before you build it.

## 2. [TUTORIAL] Running a real multi-agent Claude Code pipeline without burning your session in 20 minutes

**Thesis:** The Explore Agent eats 100k tokens and the main agent re-reads it anyway. Here's how to orchestrate parallel Claude Code workers efficiently using IPC, launchd, and scoped context files.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=8 voice=9 gap=9 effort=high

### REDDIT Version

running a real multi-agent Claude Code pipeline without burning your session in 20 minutes

here's the pattern i kept hitting: I'd spin up an Explore agent to research a codebase, it'd eat 80-100k tokens building a picture of the architecture, then hand back a summary to my main agent . which would promptly re-read half the same files to verify it. context hemorrhage on both ends.

the fix isn't smarter agents. it's scoped handoffs and real IPC between them.

**the actual problem**

Claude Code's agent tool is synchronous by default. when you launch a subagent from the main context, you're in the same session lineage. the main context window grows with every agent call. 3 research agents = 3x the token bleed before you've written a line of code.

what you actually want: workers that run independently, write structured output to disk, and let the main agent consume summaries . not re-process raw files.

**the setup i use**

1. **scoped context files** . before launching any agent, I write a tightly scoped `.context.md` to a temp directory. this file is the only thing the subagent reads at start. it defines: what question it's answering, what files it's allowed to touch, what format the output needs to be.

example:
```
# scope: auth flow analysis
question: how does session validation work in /lib/auth/?
allowed paths: /lib/auth/, /middleware/auth.ts
output format: bulleted list, max 30 lines, no code blocks
write output to: /tmp/claude-agents/auth-analysis.txt
```

the agent never goes rogue because it has explicit rails from line 1.

2. **launchd for true parallelism** . instead of spawning agents inside Claude Code, I use launchd to run separate `claude -p` (print mode) processes as background workers. each reads its own context file and writes to its own output file.

```bash
#!/bin/bash
# run_agent.sh
CONTEXT_FILE=$1
OUTPUT_FILE=$2

claude --print \
 --system "$(cat $CONTEXT_FILE)" \
 --max-tokens 4000 \
 "analyze the codebase per your context file and write output to $OUTPUT_FILE" \
 > $OUTPUT_FILE
```

wrap this in a launchd plist with `RunAtLoad false` and trigger it programmatically. you now have real OS-level parallel processes, not nested agents eating your main window.

3. **IPC via filesystem** . the main Claude Code session polls a manifest file instead of waiting on agent returns. each worker writes a status flag when done:

```
/tmp/claude-agents/
 auth-analysis.txt ← worker output
 auth-analysis.done ← empty file, presence = complete
 schema-analysis.txt
 schema-analysis.done
```

main agent checks: `ls /tmp/claude-agents/*.done` before consuming anything. no blocking, no shared context.

4. **summary injection, not file dumps** . when the main agent reads worker output, it only reads the summary files . never the source files the workers analyzed. the workers already did that work. injecting 500 lines of analyzed output vs. 5000 lines of raw source = massive difference in tokens consumed by your main session.

**what this actually looks like in a session**

i'm working on a feature that touches auth, database schema, and frontend routing. instead of one massive Explore agent chewing through all three:

- `bash run_agent.sh auth.context.md /tmp/claude-agents/auth.txt &`
- `bash run_agent.sh schema.context.md /tmp/claude-agents/schema.txt &`
- `bash run_agent.sh routing.context.md /tmp/claude-agents/routing.txt &`

all three run in parallel outside my main session. main agent waits ~2 minutes, checks for `.done` files, reads three 30-line summaries, and starts building. total tokens consumed by the main session for the research phase: under 3k.

without this: research phase alone used to push 60-80k before i'd written anything.

**the catch**

print mode (`--print`) doesn't have access to your full tool set by default. workers are best for read-only analysis tasks. for write operations, you still want them inside Claude Code proper . but with explicit tool restrictions and a tight system prompt.

also: this only works if your context files are tight. vague context = vague output = main agent asking follow-up questions that defeat the whole point. spend 5 minutes writing good context files. it pays back 10x.

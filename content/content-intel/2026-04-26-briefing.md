**the core split: is this a reasoning task or an execution task?**

reasoning tasks = novel problem solving, architecture decisions, debugging something weird, writing prompts that need to generalize across edge cases, anything where wrong output costs you real time to fix

execution tasks = applying a known pattern, filling in boilerplate, reformatting data, writing tests for code you already understand, generating variations of something that already works

Opus is for reasoning. Sonnet is for execution. Haiku is for high-volume low-stakes ops (classification, short summaries, routing logic in agentic loops).

---

**my actual decision tree:**

1. is this the first time I'm solving this problem?
 - yes → Opus
 - no, I've solved something like this before → Sonnet

2. if it fails, how long does it cost me to recover?
 - more than 30 min → Opus
 - under 30 min → Sonnet

3. is this task inside an agentic loop that runs 50+ times?
 - yes → Haiku if it's classification/routing, Sonnet if it's generation
 - no → apply rules 1 and 2

4. am I writing or editing a prompt that other AI calls will use downstream?
 - yes → Opus. always. this is highest leverage reasoning work
 - no → continue

---

writing tests for code I already wrote and understood → was using Opus. switched to Sonnet. zero difference in output quality. significant difference in cost at scale.

generating structured data (JSON schemas, column mappings, formatted outputs) where the structure is already defined → Sonnet handles this perfectly. Opus is overkill.

initial architecture design for a new system → Sonnet gave me mediocre options I had to redo. Opus on the first pass saved me the back-and-forth.

## 2. [TUTORIAL] running parallel Claude Code agents on the same repo without them fighting over ports and state

**Thesis:** Parallel Claude Code sessions are a multiplier until they all try to spin up port 3000 at once. A two-part setup — dynamic port assignment plus a shared coordination file — eliminates the collision problem entirely.
**Target sub:** r/r/ClaudeCode

### REDDIT Version

running parallel Claude Code agents on the same repo without them fighting over ports and state

parallel Claude Code sessions are genuinely a multiplier until they're not. if you've spun up two or three at once on the same repo, you've hit this: agent 1 grabs port 3000, agent 2 tries to start the dev server, everything explodes, both agents start hallucinating fixes for an error that's just a port collision.

two-part fix that actually holds in production.

**part 1: dynamic port assignment**

stop hardcoding ports in your dev scripts. instead, use a helper that finds an open port at startup:

```js
// scripts/get-port.js
const net = require('net');

function getOpenPort(start = 3000) {
 return new Promise((resolve, reject) => {
 const server = net.createServer();
 server.listen(start, () => {
 const port = server.address().port;
 server.close(() => resolve(port));
 });
 server.on('error', () => resolve(getOpenPort(start + 1)));
 });
}

getOpenPort().then(port => {
 process.env.PORT = port;
 console.log(`PORT=${port}`);
});
```

then in your package.json:

```json
"dev": "node scripts/get-port.js && next dev -p $PORT"
```

each agent session now grabs whatever port is actually free. no collisions.

**part 2: the coordination file**

ports are half the problem. the other half is agents stomping on each other's state. migrations running twice, seed scripts conflicting, agents both trying to write the same env config.

the fix is a simple `.agent-lock.json` at the repo root that each session registers to on startup:

```json
{
 "sessions": {
 "agent-1": { "port": 3001, "task": "auth-refactor", "pid": 84721, "started": "2026-04-27T14:32:00Z" },
 "agent-2": { "port": 3002, "task": "dashboard-rebuild", "pid": 84899, "started": "2026-04-27T14:35:00Z" }
 }
}
```

I add a CLAUDE.md instruction that tells each session to:
1. read `.agent-lock.json` before running any migrations or global setup
2. register itself with a unique session ID (I use the task name) and its assigned port
3. skip any operation that another session already owns

the key line in CLAUDE.md:

```
before running db migrations, seeding, or any global env setup:
read .agent-lock.json. if another session has already claimed that operation today, skip it and log why.
register your session under your task name before starting work.
```

agents are surprisingly good at following this when the instruction is explicit. vague CLAUDE.md = ignored. specific condition + specific action = followed.

**what this eliminates**

- port 3000 EADDRINUSE errors
- duplicate migration runs
- agents fighting over .env writes
- "why is the db seeded with garbage data" mystery

**what it doesn't fix**

git conflicts if two agents edit the same file. that's a workflow problem, not a tooling problem. structure your parallel tasks so they own different parts of the codebase. agent 1 owns `/app/auth`, agent 2 owns `/app/dashboard`. overlap is where things still get messy.

been running 3-4 parallel sessions on the same monorepo consistently with this setup. the coordination file feels manual but it's like 20 lines and it works. haven't had a port collision in weeks.

the reason you can't debug your multi-agent setup has nothing to do with the framework

everyone goes straight to logs. traces. LangSmith, Langfuse, whatever observability layer they bolted on. all green. agents completing. tasks marked done.

and the system is completely broken.

here's what's actually happening: your agents are hallucinating consensus on stale or wrong state. not hallucinating in the "made up a fact" sense. hallucinating in the "confident read of a shared context object that doesn't reflect reality" sense. and your traces show successful tool calls because the calls succeeded. the tool ran. the write happened. it's just writing the wrong thing with full confidence.

I've seen this pattern kill multi-agent pipelines three different ways:

**the stale handoff.** agent A finishes a subtask and writes its output to shared state. agent B reads shared state but pulls a cached version from two steps ago. both agents report success. your orchestrator marks the task complete. the output is a chimera of two different reasoning paths that never actually merged.

**the schema drift.** you update what agent A writes to the context object mid-run. agent B was written to expect the old shape. it doesn't error out. it just silently reads `undefined`, treats it as null/empty, and continues with degraded inputs. no exception. no trace flag. just quietly wrong.

**the optimistic lock problem.** two agents write to overlapping keys in quick succession. last write wins. neither agent knows the other wrote. the final state looks coherent but it's half A's reasoning and half B's, stitched together at the key level with no awareness of the conflict.

the fix isn't better logging. it's treating your shared context object like a database with actual constraints.

what works:

- version your state. every write to shared context increments a version field. every read validates it's reading the version it expects. if not, halt and surface it explicitly instead of continuing on bad state.

- make handoffs explicit and synchronous at the boundaries that matter. async everything is fine for pure tasks. but when agent A's output is agent B's input, that boundary needs a real handoff check, not just "the key exists in the object."

- write a context validator that runs before any agent consumes shared state. not a type check. a semantic check. does the state actually make sense given where we are in the workflow? I've been using a lightweight Claygent-style prompt that takes the current state object and the expected workflow stage and just asks "does this state match what should be true at this point?" if it says no, stop. don't let the downstream agent run on garbage.

the framework is fine. the model is fine. the observability tool is showing you exactly what happened. what it can't show you is whether what happened was correct given what should have been true.

treat shared state like it will lie to you. build the checks that catch it when it does.

the symptoms: Claude would nail a task on Monday, then do something completely inconsistent by Thursday. same project. same CLAUDE.md. different outputs. I was blaming model behavior, blaming context window limits, blaming compaction. filed it under "just how it is."

the actual problem: I was treating CLAUDE.md like a config file and memory like a log. neither of those mental models is right.

here's what I was doing wrong:

**CLAUDE.md was a graveyard**
I kept adding rules every time something broke. "never do X." "always do Y first." by month two it was 340 lines of defensive instructions written in response to past failures. Claude was reading 340 lines of trauma and trying to infer what I actually wanted. context was polluted before the session even started.

**memory files were append-only**
I had a system that wrote memories after each session. I never updated or removed them. so over time I had contradictory memories sitting side by side. "user prefers approach A" from February and "user prefers approach B" from March, both loaded. Claude was averaging them and producing C.

**I was loading everything, always**
skills, patterns, tool configs, project notes all loading regardless of what I was doing. a coding session was importing GTM workflow context. a content session was importing infra notes. the relevant signal was buried in irrelevant noise.

the fix that actually worked:

1. **CLAUDE.md down to 40 lines**. only stable invariants. not "don't do X" instructions written in the heat of the moment. if a rule exists because of one bad session, it probably doesn't belong there. real patterns surface across 10+ sessions. those get in. one-offs don't.

2. **memory files are living documents, not logs**. when a preference changes, I update the old entry. when something turns out to be wrong, I delete it. the test: can I read the memory file cold and get an accurate picture of the project? if not, it's wrong.

3. **MEMORY.md is a map, not a dump**. top-level file stays under 200 lines. links out to topic files. Claude loads the map, then fetches what's relevant. I stopped trying to preload everything and started trusting that specific context can be fetched on demand.

4. **session-scoped context stays out of persistent files**. current task, in-progress state, what I was thinking on Tuesday... none of that goes in memory. that stuff belongs in the conversation, not baked into the context that loads fresh every session.

the counterintuitive part: less context loaded consistently produces better results than more context loaded inconsistently. I kept thinking the solution was more memory, more rules, more detail. the solution was curation.

it's been six weeks since I overhauled this. context drift is basically gone. not because Claude changed, but because what Claude is reading at session start is actually coherent now.

boring fix. works every time.

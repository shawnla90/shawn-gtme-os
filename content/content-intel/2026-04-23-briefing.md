here's how it works:

**the skill does three things in order:**

1. audits open tasks (reads the current TodoWrite state, flags anything incomplete)
2. stashes a session snapshot to a markdown file with timestamp, active context, what was decided, what's still open
3. blocks exit and shows you a human review checklist before the session actually closes

the block is the important part. without friction, you'll skip the review every time.

---

**the actual skill file** (lives in `~/.claude/skills/close.md`):

```
# /close . Session Close Protocol

When invoked, perform these steps in order:

## 1. Task Audit
- Read current todo state
- List any incomplete items
- Flag anything marked in_progress

## 2. Stash Context
Write a session snapshot to ~/sessions/YYYY-MM-DD-HH-MM.md with:
- What was being built
- Key decisions made this session
- Open questions or blockers
- Files changed (git diff --stat)
- Next session starting point

## 3. Review Gate (BLOCKING)
Do NOT end the session until the user confirms each:
- [ ] No in-progress tasks left dangling
- [ ] Output was reviewed by human eyes, not just run
- [ ] Session snapshot written
- [ ] Any prompts or formulas worth keeping are saved somewhere

Show this checklist. Wait for explicit confirmation before closing.
```

```markdown

## what I was building

## decisions made

## still open
- HTTP API step needs testing on edge case
- haven't validated the fallback logic

## files touched
[git diff --stat output]

## next session starts here
```

there's a whole genre of CLAUDE.md templates floating around that look incredibly thorough. role definitions, output formats, a philosophy section. some of them are genuinely impressive as writing.

they also don't work for your codebase.

I've built 50+ skills in production on a monorepo and my CLAUDE.md has almost nothing in common with the viral templates. here's what's actually in mine and why each section exists.

**stack declaration first, not last**

most templates bury this. mine opens with it.

```
# Stack
- Runtime: Node.js 20, zsh scripts
- AI: Claude API (claude-opus-4-6 default, haiku for fast ops)
- Storage: SQLite via better-sqlite3
- Infra: Railway, Vercel
- Key integrations: HubSpot, Apify, Instantly, Clay webhooks
```

why this matters: Claude will make different architectural choices depending on what it thinks you're running. if your CLAUDE.md doesn't say you use Railway, it might suggest Heroku patterns or skip the Dockerfile entirely. declare the stack early so every response is grounded in your actual constraints.

**your naming conventions, not generic ones**

generic templates say "use descriptive variable names." that's useless.

mine says:

```
# Conventions
- Skills live in skills/<skill-name>/index.js
- Each skill exports a default async function(args, context)
- MCP servers in mcp-servers/<name>/index.js
- DB modules in lib/db/<name>.js, always export named functions
- No default exports outside of skills
```

this is the only way Claude stops inventing new file structures every session. took me getting burned twice before I wrote this down.

**your failure modes, documented**

this is the section nobody tells you to write. mine has grown entirely from production incidents.

```
# Known failure modes . don't repeat these
- DO NOT use fs.promises.readFile without checking if path exists first. I have broken deploys over this.
- DO NOT install new npm packages without checking if a lighter built-in alternative exists. bundle size matters.
- DO NOT use async/await in top-level scripts without wrapping in an IIFE or main(). Railway will hang.
- When writing to SQLite, always check if table exists before insert. schema migrations are manual here.
```

none of that is in any template. that's just pain I encoded so I stop re-experiencing it.

**output conventions that match your actual workflow**

I use Claude Code inside a terminal most of the time. my CLAUDE.md reflects that:

```
# Output behavior
- Default to concise responses. I read diffs, not summaries.
- When writing new skills, output the full file. don't truncate.
- When unsure about a path or integration, ask before writing. don't guess and create files.
- Commit messages: imperative mood, under 72 chars, no co-author lines unless I ask.
```

the "don't truncate" rule exists because I got burned with half-written skill files that looked complete until they weren't. the "ask before writing" rule exists because Claude was creating new db modules when I already had one that did the same thing.

**what to leave out**

don't put your philosophy in CLAUDE.md. it reads nice, doesn't change behavior.

don't write "be concise" without defining what concise means for your use case. my definition is: one sentence summary, then the diff, then stop. yours might be different.

don't copy paste someone else's role definition. "you are an expert senior software engineer" adds nothing if Claude is already loading context from your actual codebase.

---

the tldr: your CLAUDE.md should read like the onboarding doc you'd write for a contractor who had one day to get productive in your repo. not a philosophy statement. not a template. a map of your specific terrain.

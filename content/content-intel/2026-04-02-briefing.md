### REDDIT Version

how I run 4 Claude Code sessions in parallel without losing my mind

been doing this for a few months now and it's the thing that made me actually ship faster instead of just feeling busy.

the tldr: split work by concern, not by task. most people open multiple sessions and run the same stuff in parallel. that's not the unlock. the unlock is treating each session like a specialist that owns a specific layer of your system.

**how I split sessions**

typically running:
- session 1: orchestrator. reads the full context, breaks work into tasks, writes the plan
- session 2: builder. executes a specific module or feature. no context from other sessions
- session 3: reviewer/tester. only sees diffs and acceptance criteria
- session 4: docs or cleanup. runs after the builder session closes work

sounds overly formal but takes 5 minutes to set up once you have the pattern.

**keeping context clean**

this is where most parallel setups fall apart. Claude Code sessions share nothing by default, which is actually what you want, but you have to be intentional about it.

what I do:

1. CLAUDE.md is the single source of truth for project-level context. every session reads it on init. I keep mine tight: tech stack, current sprint focus, conventions, what NOT to touch. if it's not in CLAUDE.md, the session doesn't know it.

2. worktrees for true isolation. `git worktree add ../project-feature-x feature/x` gives each session its own branch and working dir. no file conflicts, no state bleed between sessions.

3. task files for handoffs. when the builder session is done, it writes a `tasks/handoff-[feature].md` with: what it built, what it skipped, open questions. the reviewer session picks that up instead of trying to re-derive context from code.

**reviewing fast enough to ship**

this is the part nobody talks about. if you're running 4 sessions and reviewing each one line by line, you're not going faster.

my review flow:
- git diff per worktree before merge
- I write acceptance criteria in plain english before the session starts, not after
- reviewer session gets: the diff, the acceptance criteria, and one question: "does this match the spec? list anything that doesn't."
- if it passes, I merge. if it doesn't, I give the builder session a focused fix prompt, not a full re-run

whole review loop takes 10-15 minutes per feature when the spec was tight going in.

**the actual failure mode**

it's not context pollution or merge conflicts. it's vague task definitions.

tight scope + clear acceptance criteria = parallel setup works. fuzzy prompts + sessions figuring it out = 4x the mess to clean up.

before I start any session now I write two things: what done looks like, and what out of scope looks like. that second one matters as much as the first.

step 2: enrichment loop

I keep a `scripts/enrich.py` that hits APIs directly based on what's missing. Hunter for email patterns, Apify for LinkedIn headcount data, Semrush for traffic estimates. Claude Code lets me iterate on the enrichment logic without rebuilding a table every time I change a column.

```python
def enrich_company(domain):
 results = {}

 hunter_resp = requests.get(
 "https://api.hunter.io/v2/domain-search",
 params={"domain": domain, "api_key": HUNTER_KEY}
 )
 if hunter_resp.status_code == 200:
 data = hunter_resp.json().get('data', {})
 results['email_pattern'] = data.get('pattern')
 results['employee_count'] = data.get('organization', {}).get('linkedin', {}).get('employees')

 # update row status
 client.table('companies').update({
 **results,
 'status': 'complete' if results else 'failed'
 }).eq('domain', domain).execute()
```

failed rows stay `status: 'pending'` and the loop picks them up on the next run. no credits wasted. no table rebuild.

step 3: scoring in SQL

this is the part that changed how I think about scoring. instead of Clay columns with waterfall logic that breaks when you move rows around, it's a view:

```sql
CREATE VIEW scored_companies AS
SELECT
 company_name,
 domain,
 employee_count,
 monthly_traffic,
 CASE
 WHEN employee_count BETWEEN 50 AND 500 THEN 20
 WHEN employee_count BETWEEN 500 AND 2000 THEN 10
 ELSE 0
 END +
 CASE
 WHEN tech_stack ILIKE '%HubSpot%' THEN 15
 ELSE 0
 END +
 CASE
 WHEN monthly_traffic > 10000 THEN 10
 ELSE 0
 END AS icp_score
FROM companies
WHERE status = 'complete';
```

change your ICP criteria? update the view. the data stays, only the logic changes. I've tweaked scoring mid-run on 50k row jobs without touching a single row.

step 4: push scored results back to Sheets

```python
scored = client.table('scored_companies')\
 .select('*')\
 .order('icp_score', desc=True)\
 .execute()

headers = list(scored.data[0].keys())
values = [[row[k] for k in headers] for row in scored.data]

output_sheet.clear()
output_sheet.update('A1', [headers] + values)
```

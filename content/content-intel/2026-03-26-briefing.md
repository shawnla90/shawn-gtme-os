I've seen a lot of "I hit my limit again" posts and almost none of "here's what I actually changed." so here's the breakdown from running GTM agents (Clay enrichment, CRM writes, LinkedIn scraping loops) for the past few months.

**1. context bloat is your biggest leak and it's invisible**

CC keeps the full conversation in context every turn. if you're 80 messages into a session debugging a Python script, every new message carries all 80. you don't feel it happening but your token burn is 3-4x what it needs to be by the end.

fix: `/clear` more aggressively than feels right. I now clear after every completed subtask, not at the end of a session. task done → `/clear` → next task. feels weird at first. burns way less.

**2. you're using opus for tasks that don't need opus**

this one hurt when I realized it. I was running sonnet-level work on opus just because the default was there. file reads, quick edits, search tasks... these don't need the most capable model. they need a fast, cheap model.

in your settings you can route by task type. I keep opus for architectural decisions and new feature builds. sonnet for implementation. haiku for anything repetitive or read-heavy. if you haven't done this yet it's the highest ROI 10 minutes you'll spend.

**3. CLAUDE.md files that are way too long**

everyone loads their CLAUDE.md with everything. every rule, every preference, every reference doc path. that thing gets injected at the start of every session and into a lot of turns.

I trimmed mine to the rules that actually change behavior. anything that's "nice to have context" got moved to topic files that I reference explicitly when relevant. my main CLAUDE.md went from ~400 lines to under 80. meaningful burn reduction on long sessions.

**4. reading files you don't need to read**

before I got into the habit of using Glob and Grep first, I was having CC read full files to find one function. a 400-line file read when you needed 8 lines is waste.

now: Grep for the specific pattern first, read the file with offset+limit if I need more context. took about a week to build the reflex. the tool is there, just have to use it.

**5. not using subagents for isolated research**

this is the one that surprised me most. when you offload a research task to a subagent, that work happens in a separate context. the main session doesn't carry the token weight of all that exploration.

before I learned this, I'd do all my research inline, burn 15k tokens going back and forth on a thing, then do the actual work. now research goes to a subagent, the result comes back clean, main session stays tight.

---

none of these are complicated. they're just habits. and once the habits are in, you stop hitting limits mid-session on a build you care about.

happy to go deeper on any of these if useful. the model routing one especially has a lot of nuance depending on what you're building.

you read about async/await. you do the tutorial. you nod along. none of it sticks because you've never been *personally betrayed* by a race condition at 11pm trying to figure out why your UI is showing stale data. the knowledge has no hooks. it floats.

you ship something. it breaks in a specific way. you now have a *reason* to understand the thing that broke. the learning is motivated, targeted, and it sticks because you remember exactly the moment the bug bit you.

- why server components can't use useState (because it hurt me personally)
- what useEffect cleanup actually does (because I had a memory leak that made my tab crawl)
- why you don't just throw async in a forEach (because I lost 3 hours to silent failures)

none of those lessons came from a course. they came from specific failures with specific consequences.

the "learn fundamentals first" crowd is optimizing for feeling competent. the vibe coding crowd is optimizing for getting something real in front of users. those aren't equal goals.

vibe coding gives you feedback. fast. on real problems. with stakes.

claude code effort levels actually explained (for people who build, not just chat)

been running Claude Code hard for a couple months across agentic GTM workflows. learned the expensive way that effort level matching is the difference between efficient builds and burning your daily limit before lunch.

here's the breakdown I wish existed on day one.

---

**the five effort tiers**

**tier 1 → quick lookup / single-file edit**
prompt it like you'd ask someone across the room. "change this function to return a dict instead of a list." "what does this column formula do?" no context loading, no planning. these cost almost nothing. don't overthink the prompt.

**tier 2 → standard bounded task**
single feature, clear output shape, one or two files in scope. "add retry logic to this API call." "write a regex that matches HubSpot deal stage names." this is where most daily work lives. let the tools run, keep the scope explicit.

**tier 3 → multi-file / multi-step**
now it needs to understand dependencies before acting. "refactor the enrichment module to support fallback providers" lands here. it'll read several files, reason about architecture, then build. roughly 3-5x the token cost of tier 2. worth it when the task actually needs that reasoning. not worth it when it doesn't.

**tier 4 → agentic with subagents**
you're launching sub-processes. parallel tool calls, specialized agents spinning up underneath. this compounds fast. a single pipeline run in my stack can touch 4-5 external APIs, multiple file reads, a Claude analysis pass, then a CRM write. one command, 15-20 tool calls underneath. budget accordingly.

**tier 5 → extended thinking + parallel agents**
full reasoning mode on a genuinely hard problem with agents running in parallel. I use this for building new skills from scratch or debugging something that's been wrong for three hours. the token hit is real. don't use this for a task that tier 2 can handle.

---

**the actual mistake people make**

throwing a tier 5 prompt at a tier 1 task.

"make this better" on a 30-line function will spiral. Claude loads context, reads related files, considers architecture, generates alternatives. you wanted a variable rename and got a 40-message thread.

the flip side is worse: tier 1 prompting a tier 4 task. "fix the pipeline" with zero context. it'll hallucinate scope, touch the wrong files, and you'll spend more tokens correcting it than if you'd been specific upfront.

---

**for agentic workflows specifically**

pattern I follow across Clay-to-HubSpot enrichment builds:

- prospecting table builds → tier 2. bounded, repetitive, known output shape
- new enrichment provider integration → tier 3. needs to understand existing column structure first
- pipeline debugging (data dropping between systems) → tier 4. it has to trace records across multiple tools and find where they're falling off
- building a new skill from scratch → tier 3-4 depending on complexity. give it a reference implementation to mirror and scope it tight before it starts

---

**the /fast toggle**

`/fast` switches to faster output, same model. it's not a capability downgrade, it's latency optimization. I use it for tier 1-2 tasks when I'm iterating. turn it off when you need the model reasoning through something with real complexity.

---

**the rule I actually follow**

before sending a prompt: what's the minimum context this task needs?

if the answer is "one file and a clear output spec" → keep it tight.

if the answer is "it needs to understand how three systems talk to each other" → give it that context explicitly upfront. don't make it discover scope through trial and error.

explicit scope = predictable token cost.
vague scope = Claude exploring, which is sometimes what you want and usually what you didn't budget for.

---

if you're hitting limits mid-session, the culprit is almost always mismatched effort tier, not the task itself. scope it right and the same work costs noticeably less.

happy to share specific prompting patterns for any of these tiers if it's useful.

### REDDIT Version

how I run parallel Clay + Claude Code sessions without burning my weekly limit in one go

been running 3-4 Claude Code agents in parallel most mornings. it's genuinely the fastest I've ever moved on complex builds. but the first week I tried this, I burned through 40% of my weekly limit before noon on a Tuesday and had basically nothing to show for it.

here's what was going wrong and how I fixed it.

**the core problem: context bleed + wrong effort levels**

when you spin up multiple agents against the same repo, each one is loading context independently. if you're not isolating what each agent is allowed to touch, they'll all pull in the same large files, re-read the same docs, and rack up tokens doing redundant orientation work.

on top of that, I was sending every task at the same effort level. debugging a small formula bug doesn't need the same token spend as architecting a new enrichment flow. I was treating everything like a haiku-level task and getting opus-level bills.

**the setup I run now**

1. worktrees for isolation

before I start a parallel session I run:

```bash
git worktree add ../project-agent-a -b agent-a
git worktree add ../project-agent-b -b agent-b
```

each agent gets its own branch, its own directory. they can't step on each other's files mid-execution, and when one makes a mess I just blow away the worktree. this alone cut my context-bleed problem by a lot.

2. CLAUDE.md scoping

each worktree gets a lean CLAUDE.md that tells the agent exactly what it's responsible for. I don't let them read the full repo structure unless I explicitly ask. something like:

```
You are working on the Clay enrichment layer only.
Relevant files: /lib/clay/, /config/clay.yaml
Do not touch: /lib/hubspot/, /scripts/, /skills/
Read these first: /lib/clay/README.md
```

short, specific, constrained. agents that know what to ignore spend way less time being curious about things they shouldn't touch.

3. effort routing before I even type the prompt

I have a rough mental checklist before I send anything:

- is this a lookup / small fix / one file? haiku via API, not Code
- is this a multi-file refactor or a new feature? Code, but with `--model sonnet`
- is this system design or something where I genuinely need the best reasoning? opus, and I'm conscious I'm spending budget on it

the biggest waste I see people do (including past me) is sending "does this look right?" questions to a full Code session. that's a haiku API call at most.

4. task queuing instead of firing everything at once

I keep a simple text file called `agent-queue.md`. when I'm building, I add tasks to it instead of opening a new agent immediately. once I have 3-4 queued up that are genuinely independent (different files, different concerns), I spin up the worktrees and start them in parallel.

if tasks touch overlapping files, they go sequential. parallel only when there's true isolation.

**what this looks like in a real morning**

say I'm working on a Clay table that pulls LinkedIn data, enriches it, scores it, then pushes to HubSpot.

I'll run:
- agent A: build out the Apify scraper integration and normalize the output schema
- agent B: write the scoring logic + Clay formula column definitions
- agent C: handle the HubSpot property mapping and push logic

these are genuinely isolated. A's output feeds B and C eventually, but they can each build to a contract I define upfront (the schema shape) without needing each other's live code.

total parallel session, maybe 90 minutes, three things done. if I'd run them sequentially with no worktree discipline, same outcome but 4+ hours and 2x the token spend from context re-loading.

**the thing that actually changed my usage**

I stopped treating Claude Code like a chat interface and started treating it like a CI job. you define the scope, you define the inputs, you fire it, you review the output. you don't hover and iterate mid-task unless something clearly breaks.

the agents are fast when you get out of their way. the limit burn happens when you stay in the loop on every step.

happy to share the agent-queue.md template or the CLAUDE.md scoping pattern I use if anyone wants it. drop a comment.

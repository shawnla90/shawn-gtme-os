the fix I landed on: at the start of every session, I tell Claude Code to maintain a `DECISIONS.md` file in the project root. every meaningful decision it makes gets a one-liner entry. timestamp, what was decided, why.

```
## 2025-07-10

- [14:32] chose SQLite over postgres for enrichment cache . no infra overhead, reads are fast enough for 10k rows
- [14:47] skipping email validation on import step . handled downstream in the send queue
- [15:10] using pagination cursor instead of offset . offset was drifting on concurrent writes
```

not a journal. not a summary. just the decisions, compressed.

here's why it works: when compaction happens, the context window clears a lot of the conversation history but the file system stays intact. Claude Code can re-read `DECISIONS.md` at any point and reconstruct the reasoning chain without me having to re-explain everything. it's essentially writing its own memory layer as it goes.

```
After each meaningful implementation decision, append a one-line entry to DECISIONS.md:
- [timestamp] decision . reason
Keep entries short. This file is your memory across context resets.
```

the log also becomes useful for me. when I come back to a project two days later I can read 20 lines and understand the current state better than reading the code.

## 7. [PSA] your 'private project' built with Claude Code probably isn't private. three security holes I see in almost every vibe-coded app.

**Thesis:** A private GitHub repo doesn't make your app secure — exposed Supabase anon keys, hardcoded credentials, and no RLS are endemic in AI-built apps and most builders don't know until something breaks.
**Target sub:** r/r/vibecoding
**Scores:** viral=8 voice=8 gap=7 effort=low

### REDDIT Version

your private GitHub repo means nothing if your app is leaking credentials.

I've reviewed probably 30+ vibe-coded apps in the last few months . people sharing builds in Discord, asking for feedback, doing demo calls. three problems show up in almost every single one. and none of them get caught until something breaks (or someone malicious notices first).

---

**hole #1: your Supabase anon key is public**

the anon key is designed to be public. Supabase docs even say so. but that's only safe if your Row Level Security policies are actually doing the work. most vibe-coded apps skip RLS entirely or leave it misconfigured.

Claude Code (and most AI assistants) will generate working Supabase code fast. the problem is "working" and "secure" aren't the same thing. if you have a table with no RLS policy, any authenticated user can query every row. your anon key just became a skeleton key.

check yours right now:

```sql
select schemaname, tablename, rowsecurity
from pg_tables
where schemaname = 'public';
```

if rowsecurity is false on tables that hold user data, you have a problem.

**hole #2: .env files committed to git**

I know, I know. everyone knows not to do this. and yet.

```
git log --all --full-history -- .env
git log --all --full-history -- "**/.env*"
```

run those two commands. if you see commits, your secrets are in the history even if you deleted the file later. `git rm` doesn't fix this. you need to rotate the keys and optionally rewrite history with git filter-repo.

the sneaky version: Claude Code generates a working .env.example, you rename it to .env, and you commit it because it's "just the example." except you filled in the real values first.

**hole #3: hardcoded credentials in client-side code**

this one is specifically brutal with React/Next.js apps. when you ship to Vercel or Netlify, anything that isn't prefixed with NEXT_PUBLIC_ (or equivalent) stays server-side. but AI-generated code sometimes puts secrets directly in components or utility files that get bundled into the client.

open devtools on your own deployed app. Sources tab. search for your API key prefixes . sk-, pk_, anything that looks like a token. if you find it, so can anyone else.

---

quick audit checklist if you want to be thorough:

- `grep -r "SUPABASE_SERVICE_ROLE" ./src` . this key should never appear in frontend code
- `grep -r "process.env" ./src/components` . anything here is worth reviewing
- check your Supabase dashboard → Authentication → Policies, every table that holds sensitive data needs at least one policy
- `git secrets --scan-history` if you want to go deep

---

none of this is Claude Code's fault. the models generate what you ask for, and "secure by default" is rarely the ask when you're vibing through a build at 2am. but these three gaps are endemic enough that I think every builder shipping something with user data should do a 30 minute audit before they share the link.

the private repo is a false ceiling. your threat surface is the running app, not the source code.

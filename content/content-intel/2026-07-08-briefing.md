here's what i changed:

**1. write-confirmation gates**

before any destructive operation, the agent now has to write a confirmation file to disk with the exact operation it's about to run, row count affected, and a timestamp. i read that file before execution continues. i added this as a mandatory step in the tool instructions, not as a request.

```
# in the tool system prompt
before executing any DELETE, DROP, TRUNCATE, or UPDATE:
1. write a file called `.pending_op.json` to the working directory
2. include: operation_type, table_name, affected_row_estimate, where_clause, timestamp
3. halt and wait. do not proceed.
```

**2. dry-run by default**

every destructive script now has a `--dry-run` flag baked in and the agent is instructed to run dry-run first, show me the output, and wait for explicit `--confirm` before the real run. i was doing this manually sometimes. now it's in the instructions so i can't forget.

**3. separate read and write tool contexts**

this is the one that changed how i structure sessions the most. read operations and write operations now live in separate tool contexts. the agent doing research and the agent doing mutations are different invocations with different permissions in the instructions. i don't let a research pass accidentally have write access because "well i might need it later."

**4. no summary acceptance without artifact**

the rule now: if an agent reports on what it did, it has to produce an artifact. a log file, a diff, a count, something i can verify independently. "no changes made" is not an acceptable terminal output without a record i can audit. i added this line verbatim to my base system prompt:

```
never summarize an operation without producing a verifiable artifact.
a log file, row count before/after, or diff output counts. 
a text summary alone does not count.
```

what i didn't do: i didn't add a bunch of confirmation prompts for every little thing. that kills the workflow. i also didn't switch to a "safer" model or avoid agentic tasks. the answer isn't fewer agents, it's better rails.

the mistake wasn't claude's. the mistake was mine for not building the guardrails before handing it the keys.

lost data is recoverable if you have backups (i did, but the restore took four hours and a lot of coffee). the harder thing to recover is the assumption that the agent's summary is a reliable source of truth.

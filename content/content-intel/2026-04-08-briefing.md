so I started pre-compiling the schema into a context wiki.

**what a context wiki actually is**

it's a markdown file that lives next to your project. not a README. not docs. a dense, compressed representation of everything Claude would otherwise have to discover by reading through your table.

mine looks like this:

```md
# TABLE: ICP Outbound Enrichment v3

## Column Registry
- person_li_url_raw: input. scraped from Apollo. never enriched, source of truth.
- person_linkedin_enriched: output of Clay LinkedIn enrichment. has title, company, headline.
- company_domain_cleaned: formula column. strips www, strips protocol. source for all domain-based enrichment.
- company_techstack_raw: HG Insights enrichment output. JSON. parse with formula before using.
- icp_score: formula. weighted sum of 6 signals. see scoring logic below.

## Enrichment Dependencies
company_domain_cleaned --> [Clearbit, HG Insights, BuiltWith, Firecrawl]
person_li_url_raw --> [LinkedIn enrichment] --> person_linkedin_enriched
person_linkedin_enriched.title --> job_level_normalized (formula)

## Scoring Logic
icp_score = (emp_band * 0.3) + (tech_match * 0.4) + (growth_signal * 0.2) + (geo_match * 0.1)
emp_band: 1-5 scale, mapped in emp_band_lookup column

## Known Quirks
- HG enrichment fails ~15% on non-.com domains. check company_domain_cleaned format first.
- Firecrawl timeout on sites >50 pages. cap crawl depth at 3.
- icp_score NULL means emp_band is NULL, not a formula error.
```

that whole file is under 500 tokens. Claude loads it at session start and already knows the table.

**how to build yours**

1. open a fresh Claude Code session on your table project
2. dump your column structure, formula logic, and enrichment setup into one conversation
3. ask Claude to compress it into a context wiki in the format above
4. save it as `[table-name]-context.md` in your project root
5. add to your CLAUDE.md: `always read [table-name]-context.md at session start`

takes 20 minutes the first time. every session after starts with Claude already oriented.

**the actual difference**

before: first 3-4 prompts in every session were me re-explaining what `company_domain_cleaned` feeds, why there's a raw vs. enriched version of the LinkedIn URL, what icp_score nulls actually mean.

after: first prompt is actual work.

the token drop is real but the bigger win is that Claude stops making confident wrong assumptions. it used to hallucinate column relationships because it was pattern-matching from incomplete context. once the wiki exists it just... knows.

**what goes in, what stays out**

worth including: column registry with types and purpose, enrichment dependency map, formula logic for anything non-obvious, known failure modes and edge cases.

not worth including: row counts, sample data, campaign names, anything that changes week to week.

the goal is a stable compressed map of the table's structure and logic. not a snapshot of current state.

one file per table, built once, loaded in under 500 tokens. if you're running repeated Clay sessions without this you're burning context budget on re-discovery instead of actual building.

**the goal**

session-per-contact enrichment. one agent session per HubSpot contact ID, the agent runs enrichment across multiple tool calls, and every action gets logged back to the contact timeline as a note. plus a streaming UI so you can watch it work in real time.

**how session management actually works**

the SDK's `ClaudeSDKClient` (Python) is an async context manager. each `client.query()` call on the same client instance continues the same session automatically. you get the session ID off `ResultMessage.session_id` on every result message.

for session-per-contact the pattern is: spin up a new client per contact, run the enrichment query, capture the session ID if you need to resume later.

```python
async with ClaudeSDKClient(
 options=ClaudeAgentOptions(
 mcp_servers={"hubspot": hubspot_mcp_config},
 permission_mode="bypassPermissions",
 max_turns=15
 )
) as client:
 await client.query(
 f"Enrich contact {contact_id}. Pull job title, company size, "
 f"verify email, then log a note on the contact summarizing what you found."
 )
 async for message in client.receive_messages():
 if message.type == "result":
 session_id = message.session_id # store for resume
 yield format_sse(message)
```

the agent has HubSpot access via MCP and handles the CRM writes itself. I'm not building a custom relay tool - I'm just telling it what to do and the MCP handles read/write. cleaner than I expected.

**the SSE part**

the SDK does not give you raw SSE. streaming is an async generator. if you want a browser-facing SSE endpoint, you wrap the generator yourself:

```python
async def enrichment_stream(contact_id: str):
 async with ClaudeSDKClient(...) as client:
 await client.query(f"Enrich contact {contact_id}...")
 async for message in client.receive_messages():
 yield f"data: {json.dumps(message.model_dump())}\n\n"

@app.get("/enrich/{contact_id}")
async def enrich(contact_id: str):
 return StreamingResponse(
 enrichment_stream(contact_id),
 media_type="text/event-stream"
 )
```

works. frontend subscribes with EventSource, gets tool call progress in real time as the agent reasons through each enrichment step.

**what broke**

two things.

first: long-running sessions and connection drops. if enrichment takes 45+ seconds (LinkedIn + multiple tool calls) the HTTP connection drops depending on your server config. added proper timeout settings on the FastAPI side and reconnect logic client-side. on reconnect, pass the stored session ID back with `ClaudeAgentOptions(resume=session_id)` and the session picks up mid-run. session state persists locally at `~/.claude/projects/<cwd>/<session-id>.jsonl` so this actually works.

second: the agent is enthusiastic with HubSpot API calls. with `max_turns=15` it was making 10-12 calls per contact (reads + writes). at volume that compounds fast. dropped `max_turns` to 8 and tightened the prompt to be more surgical about what it reads. call count dropped significantly.

**what's actually solid right now**

session management is clean. the resume pattern is the part worth paying attention to - tested it by killing the process mid-run and resuming with the stored session ID. picked up exactly where it left off. for enrichment workflows that can die on you mid-batch, that's a real unlock.

for a HubSpot sidebar extension or single-contact enrichment UI this is basically production-ready. the async generator wraps cleanly into SSE, and the MCP integration means you're not writing glue code to talk to the CRM.

for batch processing at volume: don't fire 500 concurrent sessions. queue in front of it. learned that immediately.

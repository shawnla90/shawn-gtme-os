i wrote a cron that starts my Claude Code session automatically . here's why and how

most people don't know this: Claude Code's 5-hour window doesn't start from when you actually open the app. it floors to the clock hour when your first message fires.

so if you open Claude Code at 9:47 AM and send your first message, your session clock started at 9:00. you're already 47 minutes into your window before you've typed anything useful.

over a 5-day week that's potentially 3-4 hours of wasted capacity. for free.

---

**the fix is embarrassingly simple**

i wrote a launchd plist (mac) that fires a keepalive message at exactly 9:00 AM on weekdays. claude processes it, session window opens at :00:00, i get the full 5 hours.

here's the shell script (`~/scripts/claude-session-start.sh`):

```bash
#!/bin/bash
cd /Users/yourusername/your-repo
echo "session start" | claude --print > /dev/null 2>&1
```

and the launchd plist at `~/Library/LaunchAgents/ai.shawnos.claude-start.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
 "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
 <key>Label</key>
 <string>ai.shawnos.claude-start</string>
 <key>ProgramArguments</key>
 <array>
 <string>/bin/bash</string>
 <string>/Users/yourusername/scripts/claude-session-start.sh</string>
 </array>
 <key>StartCalendarInterval</key>
 <dict>
 <key>Hour</key>
 <integer>9</integer>
 <key>Minute</key>
 <integer>0</integer>
 <key>Weekday</key>
 <integer>0</integer>
 </dict>
 <key>RunAtLoad</key>
 <false/>
</dict>
</plist>
```

load it once:
```bash
launchctl load ~/Library/LaunchAgents/ai.shawnos.claude-start.plist
```

if you're on linux/WSL, the crontab version is even shorter:

```bash
0 9 * * 1-5 cd /your/repo && echo "session start" | claude --print > /dev/null 2>&1
```

---

**a few gotchas**

- `--print` flag is key. it runs claude non-interactively and exits clean. without it the process hangs.
- make sure the `cd` path matches where you actually work. claude needs a project context or it'll just spin.
- if you start work at inconsistent times, skip the cron and just add a shell alias: `alias cs='echo "session start" | claude --print > /dev/null 2>&1'` and fire it manually when you sit down.
- the keepalive message does count against your context, but it's ~4 tokens. basically nothing.

---

**why this matters more than it sounds**

i run claude code for 6-8 hours a day across multiple repos. before i figured this out i was regularly starting at :30 or :45 past the hour, which meant mid-afternoon i'd hit the window limit right when i was deep in something.

now the session pre-warms at 9:00, i open the terminal when i'm ready, and the full window is there.

small operational fix. real daily impact.

happy to share the full plist template if anyone wants it . drop a comment and i'll post it.

**the 35 build-time feature flags**

this was the first thing I checked. the source has a flag map baked into the build . not runtime toggles, compile-time. some of these explain behaviors that felt random before:

- there's a flag controlling whether parallel tool calls are allowed in a single response turn. this is why the same prompt sometimes fans out tools and sometimes executes them sequentially . different builds, different flags
- plan mode is gated behind a flag that also tracks whether the todo integration is active. they're coupled internally, not separate features
- context compression threshold is configurable via flag. the default kicks in around 60% utilization. after that, older tool results get dropped before conversation turns

the implication: if you're on Claude Max and something feels off about behavior compared to API access, it may not be your prompt. different build targets, different flag profiles.

---

**context management is tiered, not flat**

the architecture isn't "put everything in the window and hope." it's three tiers:

1. system prompt layer . CLAUDE.md files, reloaded fresh on every turn
2. active window . recent messages + live tool results
3. compressed history . older turns get summarized and pushed back

compression is lossy by design. when it runs, it rewrites prior exchanges into summaries. this is why Claude will sometimes work from a slightly off recollection of specific wording you used 40 messages ago . it's not hallucinating, it's reading a summary of a summary.

knowing this changes how you structure long sessions. put the stuff that must survive compression in your CLAUDE.md, not in early conversation turns.

---

**tool result truncation**

tool results above a hard token limit get cut before they hit the model. the threshold I found is in the source. if you're reading large files and the model consistently misses content near the end . that's the truncation, not attention drift.

the fix: use `offset` and `limit` on file reads. reading 200 lines at a time isn't slower in practice. it's more reliable because you control what lands in context.

---

**CLAUDE.md loading order**

this one's been debated in the community. the source confirms it:

1. global `~/.claude/CLAUDE.md`
2. project root `CLAUDE.md`
3. subdirectory `CLAUDE.md` (closest to working directory)

proximity wins on conflicts. if you have a rule at root and an overriding rule in `/packages/ui/`, the subdirectory wins for any work happening inside that folder.

I've been structuring my monorepo CLAUDE.md files around this for a while but it was assumption-based. good to see it confirmed in the source.

---

**permission modes are different execution paths**

auto-approve, ask, and manual don't just change what gets confirmed. they route through different internal pipelines. auto mode skips the approval prompt generation step entirely . that's part of why it's faster. the latency difference isn't just the missing dialog box, it's a shorter code path.

---

**subagent token budgets**

subagents get a capped budget separate from the main session. the default cap is lower than you'd expect for complex research tasks. this explains the "shallow result" problem . the subagent ran out of budget, not ideas.

workaround that works: give subagents tight file scope upfront. "only read files matching `*.tsx` in `/src/components`" burns budget on signal. "look at the frontend" burns it on navigation.

### REDDIT Version

how I cut Claude API costs 80% using Haiku as a gatekeeper before Sonnet

been running a content pipeline that hits Claude API dozens of times per day. sonnet costs were stacking fast. here's the routing pattern that fixed it.

**the problem**

I was sending everything to claude-sonnet-4-6. classification tasks, extraction tasks, simple reformatting, and actual generation. all sonnet. dumb.

sonnet is maybe 10x the cost of haiku per token. but I was only actually *needing* sonnet on maybe 20% of calls.

**the pattern: two-stage routing**

stage 1 . haiku decides what the request is
stage 2 . route to haiku or sonnet based on that classification

```python
HAIKU = "claude-haiku-4-5-20251001"
SONNET = "claude-sonnet-4-6"

def classify_request(prompt: str) -> str:
 """Haiku decides which model handles this."""
 response = client.messages.create(
 model=HAIKU,
 max_tokens=20,
 messages=[{
 "role": "user",
 "content": f"""Classify this task. Reply with exactly one word: SIMPLE or COMPLEX.

SIMPLE = extraction, classification, formatting, yes/no, short factual answers
COMPLEX = generation, reasoning, synthesis, creative writing, code

Task: {prompt[:500]}"""
 }]
 )
 return response.content[0].text.strip()

def route(prompt: str, **kwargs):
 task_type = classify_request(prompt)
 model = SONNET if task_type == "COMPLEX" else HAIKU
 
 return client.messages.create(
 model=model,
 messages=[{"role": "user", "content": prompt}],
 **kwargs
 )
```

the classification call itself costs almost nothing. haiku is fast and nearly free at that token count.

**what I route to haiku**

- "does this text mention a competitor?" → yes/no
- "extract the company name from this job posting"
- "is this email a reply or a new thread?"
- "categorize this support ticket: billing, technical, general"
- "does this linkedin post have a CTA?"

none of those need sonnet. I was paying sonnet rates for binary answers.

**what still goes to sonnet**

- drafting outreach copy
- synthesizing research into a brief
- reasoning through a multi-step workflow
- anything where I'd notice the difference

**the 80% number**

my pipeline before: ~95% sonnet calls
after routing: ~18% hit sonnet

cost dropped accordingly. not magic. just stopped using a sledgehammer for thumbtacks.

**why this maps to how Claude Code routes internally**

if you've watched Claude Code work, you've probably noticed it's not burning opus on every little grep or file read. there's lightweight routing happening. the same principle. match model capability to task complexity.

you can extend this pattern with a third tier . use haiku for classification, haiku for simple tasks, sonnet for complex, and add opus for genuinely hard synthesis if you need it. I haven't needed opus in the pipeline yet but the slot is there.

**one gotcha**

haiku's classification isn't perfect. you'll get edge cases where it calls something SIMPLE and the output quality visibly drops. fix: add a confidence tier. if haiku isn't sure, default to sonnet. I do this with a second token that says SIMPLE_CERTAIN, SIMPLE_MAYBE, or COMPLEX.

```python
# extended classifier
content = f"""Classify: SIMPLE_CERTAIN, SIMPLE_MAYBE, or COMPLEX.

SIMPLE_CERTAIN = obvious extraction/boolean/formatting
SIMPLE_MAYBE = seems simple but could need nuance 
COMPLEX = generation, reasoning, synthesis

Task: {prompt[:500]}"""
```

route SIMPLE_MAYBE to sonnet if you want to be safe, or tune it based on your own error tolerance.

---

**first: how hooks work**

hooks live in `.claude/settings.json` (project-level) or `~/.claude/settings.json` (global). they're shell commands that fire on specific events. PreToolUse can return exit code 1 to block the tool entirely.

basic structure:

```json
{
 "hooks": {
 "PreToolUse": [
 {
 "matcher": "Bash",
 "hooks": [
 {
 "type": "command",
 "command": "your-script-here"
 }
 ]
 }
 ]
 }
}
```

the tool input comes in via `$CLAUDE_TOOL_INPUT` as JSON. PostToolUse gets `$CLAUDE_TOOL_OUTPUT` too.

---

**the 5 event types (not 23 . the 23 is the tool targets)**

- `PreToolUse` . fires before tool execution. exit 1 = blocked
- `PostToolUse` . fires after, gets the output
- `Notification` . when Claude sends you a system notification
- `Stop` . when the session ends
- `SubagentStop` . when a subagent completes

the "23" comes from targeting individual tools . Bash, Edit, Write, Read, Glob, Grep, Agent, WebFetch, WebSearch, TodoWrite, LSP, etc. you can match any of them specifically or use `"*"` for all.

---

**the ones I actually use**

**1. block force pushes + nukes globally**

```json
{
 "hooks": {
 "PreToolUse": [
 {
 "matcher": "Bash",
 "hooks": [
 {
 "type": "command",
 "command": "echo \"$CLAUDE_TOOL_INPUT\" | jq -r '.command // \"\"' | grep -qE '(git push --force|rm -rf /|DROP TABLE|truncate /)' && { echo 'blocked: dangerous command' >&2; exit 1; } || exit 0"
 }
 ]
 }
 ]
 }
}
```

this runs on every Bash call. if Claude tries to force push or nuke something, it gets blocked with a message. Claude sees the stderr output and adjusts.

**2. audit log every file write**

```json
{
 "hooks": {
 "PostToolUse": [
 {
 "matcher": "Write",
 "hooks": [
 {
 "type": "command",
 "command": "echo \"$(date -u +%Y-%m-%dT%H:%M:%SZ) WRITE $(echo $CLAUDE_TOOL_INPUT | jq -r '.file_path')\" >> ~/.claude/audit.log"
 }
 ]
 }
 ]
 }
}
```

every file Claude writes gets timestamped in an audit log. I use this on client projects. 3 weeks of Claude Code sessions and you have a full change history that isn't git.

**3. auto-lint on file save**

```json
{
 "hooks": {
 "PostToolUse": [
 {
 "matcher": "Edit",
 "hooks": [
 {
 "type": "command",
 "command": "FILE=$(echo $CLAUDE_TOOL_INPUT | jq -r '.file_path'); [[ $FILE == *.ts || $FILE == *.tsx ]] && cd $(git -C $(dirname $FILE) rev-parse --show-toplevel 2>/dev/null) && npx eslint $FILE --fix --quiet 2>/dev/null; exit 0"
 }
 ]
 }
 ]
 }
}
```

every TypeScript edit gets auto-linted. Claude never has to be told to fix lint errors because they're already gone. always exit 0 here . you don't want to block the edit, just clean it up.

**4. desktop notification on Stop**

```json
{
 "hooks": {
 "Stop": [
 {
 "matcher": "*",
 "hooks": [
 {
 "type": "command",
 "command": "osascript -e 'display notification \"Claude Code finished\" with title \"Claude Code\" sound name \"Glass\"'"
 }
 ]
 }
 ]
 }
}
```

walk away from a long task and get a macOS notification when it's done. `SubagentStop` fires for each subagent if you're running teams . useful for tracking parallel work.

**5. cost guard (experimental)**

```json
{
 "hooks": {
 "PreToolUse": [
 {
 "matcher": "Agent",
 "hooks": [
 {
 "type": "command",
 "command": "AGENT_COUNT=$(cat /tmp/claude_agent_count 2>/dev/null || echo 0); [ $AGENT_COUNT -gt 5 ] && { echo 'agent limit reached for this session' >&2; exit 1; }; echo $((AGENT_COUNT + 1)) > /tmp/claude_agent_count; exit 0"
 }
 ]
 }
 ]
 }
}
```

this caps subagent spawns per session. rough but effective if you're on API and a recursive agent loop goes sideways.

---

**the matcher field is more powerful than the docs show**

you can match partial tool names. `"Bash"` catches Bash. `"*"` catches everything. I haven't tested regex matchers but the JSON structure supports a `"matcher"` key that appears to accept patterns.

what I haven't fully explored: chaining multiple hooks on the same event/tool. the `hooks` key is an array . you can stack commands and they run sequentially. if any PreToolUse hook exits 1, the tool is blocked.

---

**the stderr trick**

when your PreToolUse hook blocks something (exit 1), whatever you write to stderr becomes the message Claude sees. Claude then decides what to do next based on that context.

so instead of just blocking, you can guide:

```bash
echo "blocked: git push requires explicit user approval. ask the user first." >&2
exit 1
```

Claude reads that and will literally ask you before trying again. it's not just a hard stop . it's a redirect.

---

**where the config lives**

- global (all projects): `~/.claude/settings.json`
- project-level: `.claude/settings.json` in your repo root
- project-level overrides global for the same event/matcher

if you're on a team, committing `.claude/settings.json` to the repo means everyone on the project gets the same guardrails automatically.

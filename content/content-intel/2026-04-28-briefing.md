psa: claude code will push to your main branch if you let it

ran into this with two different teams in the last month. both discovered it the same way. not ideal.

here's what happens: you spin up an agentic session, give claude code write access to your repo, it does the work, and somewhere in the task chain it runs `git push origin main` because nothing told it not to. no prompt, no warning. just... done.

this isn't a claude code bug. it's a missing guard rail on your end.

three things to set before your first agentic session:

**1. branch protection on GitHub/GitLab**

go to Settings → Branches → Add branch protection rule for `main` (or `master`, or whatever you call it). enable "Require pull request before merging." that's it. even if claude code tries to push directly, it'll get rejected.

**2. set a default branch in your CLAUDE.md**

add this:

```
## Git workflow
Always work on a feature branch. Never commit directly to main. Branch naming: feat/<short-description>
```

claude code reads CLAUDE.md at session start. explicit instructions in there carry more weight than you'd expect.

**3. if you're running headless or via sdk, scope the git permissions**

if you're calling claude code programmatically, you can restrict what shell commands it's allowed to run. don't give blanket bash access if you only need file edits and test runs.

the deeper thing: agentic sessions are genuinely autonomous. claude isn't waiting for you to review each step the way you would in a chat. it's executing a plan. if your repo is configured to allow direct pushes to main, it'll treat that as a valid action.

you don't need to distrust the model. you need to configure your infrastructure like you'd configure it for any other automated process that has write access.

this is a three-minute fix that prevents a very bad monday morning.

## 9. [TUTORIAL] how I wire multiple Claude Code agents together so they share context without blowing up the session

**Thesis:** Most multi-agent Claude Code setups fail because each session is an island. Routing shared state through a central memory file and using a coordinator agent pattern solves the handoff problem without custom infra.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=7 voice=9 gap=8 effort=medium

### REDDIT Version

most multi-agent Claude Code setups I've tried fail for the same reason: each agent session is a complete island.

agent A does research, hands off a summary to agent B, and agent B starts fresh with no real understanding of what agent A decided, why it made those choices, or what dead ends it already hit. the handoff is lossy by default.

here's what actually works: a coordinator agent pattern with a shared memory file as the backbone.

**the core setup**

you need three things:
1. a `shared_state.md` file that every agent reads at startup and writes decisions back to
2. a coordinator agent whose only job is routing tasks and updating state
3. worker agents that read state on init, do a scoped task, then write results back before exiting

**shared_state.md structure**

I keep mine simple:

```
# Current Task
[what we're building right now]

# Decisions Made
- [date] decided X because Y
- [date] ruled out Z because it conflicted with X

# Open Questions
- [question that needs resolution before proceeding]

# Handoff Notes
[what the next agent needs to know immediately]
```

that last section is the one that changed everything. it's not a summary of what happened. it's specifically what the next agent needs to know to not repeat work or go down a dead end.

**coordinator agent pattern**

the coordinator doesn't do the actual work. it:
- reads shared_state.md
- figures out what task is next based on current state
- spawns a worker agent via the Agent tool with a scoped prompt that includes the relevant state
- waits for the worker to finish
- reads the worker's output
- updates shared_state.md
- decides if there's another task to route or if we're done

the CLAUDE.md for the coordinator session looks like this:

```
# Coordinator Agent

Your only job is orchestration. You do not write code, do not make design 
decisions, do not research.

Before every action, read /path/to/shared_state.md.
After every worker completes, update shared_state.md with decisions made 
and handoff notes.

Worker agents are scoped to a single task. Give them exactly what they 
need from state, nothing more.
```

**worker agent prompt structure**

when the coordinator spawns a worker, I pass state explicitly in the prompt rather than having the worker re-read the whole file. keeps the worker focused:

```
Context from shared state:
- Current task: [X]
- Decisions already made: [relevant ones only]
- What NOT to do: [dead ends from handoff notes]

Your task: [specific scoped thing]

When done, write your output to /path/to/shared_state.md under 
"Handoff Notes" and summarize decisions made under "Decisions Made".
```

**why this works**

the file is the source of truth, not any individual session's context window. each agent gets exactly the context it needs, not a full dump of everything that's happened. and because decisions get written down with reasoning, a new worker agent actually understands *why* the codebase looks the way it does, not just what it looks like.

**the failure mode this prevents**

without shared state, agent B sees the output of agent A and often tries to "fix" or "improve" things that agent A intentionally did a specific way. I've watched agents undo each other's work in a loop. the `Decisions Made` section with reasoning is what breaks that loop.

**one gotcha**

if you let agents write to shared_state.md freely without structure, it becomes noise fast. I learned this after 4 agents dumped raw output into the file and it became unreadable. enforce the structure in every agent's CLAUDE.md and in the coordinator's spawning prompt. make it explicit, not optional.

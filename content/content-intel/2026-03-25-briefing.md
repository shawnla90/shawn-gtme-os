the hidden token overhead in Claude Code nobody measures (and how I cut mine by 3x)

spent two weeks wondering why my Claude Code sessions were burning out mid-task on a 50k+ line monorepo. kept blaming the model. wrong diagnosis.

the problem is context accumulation. here's what's actually happening before you type a single character:

**what gets injected at session start:**
- full system prompt (tool schemas, safety instructions, formatting rules)
- every CLAUDE.md in your project tree, recursively
- git status snapshot
- MEMORY.md if you have auto-memory configured
- any system-reminder content from MCP servers

on my setup that's ~8-12k tokens before the first message lands. not huge alone. but it compounds.

**the compounding problem:**

every file you read stays in context. every tool result. every failed attempt before the right one. by the time you're 30 messages in on a complex refactor, you've got 60-80k tokens of accumulated context from reads, writes, grep results, and correction loops. the model isn't degrading . you're just running out of room.

**what actually helped:**

1. **audit your CLAUDE.md files**. mine had a full table of sqlite queries, architecture diagrams, machine topology docs. beautiful reference material. terrible context injection. moved everything non-essential to separate docs and added "read this file when relevant" instructions instead of embedding the content. dropped my startup injection by ~40%.

2. **subagents for isolated research**. when I need to explore a part of the codebase I haven't touched, I spin a subagent with a narrow prompt instead of reading files in the main session. the subagent burns its own context. main session stays clean.

3. **/compact is not a last resort**. I run it proactively after completing a discrete task within a session, before starting the next one. not when I'm desperate . as a workflow checkpoint. Claude Code's summary generation is good enough that you lose almost nothing important.

4. **glob and grep before read**. I used to cat files I was pretty sure had what I needed. now I grep first. a grep result is 3 lines. a file read is 200. over 10 lookups that's the difference between 30 tokens and 2000.

5. **don't fight in session, close and restart**. if I'm 40+ messages into a debugging loop and not converging, the right call is /handoff, close, new session with just the relevant error and context. fresh context + fresh framing usually cracks it in 5 messages.

**the thing that surprised me:**

MCP server instructions are injected per-tool-call in system-reminder tags. if you have 15 MCP servers configured (I do), those instructions accumulate in the context across the session. I didn't realize how much space the posthog MCP docs alone were taking until I looked at it.

not saying cut your MCPs. saying know what you're carrying.

**quick formula I use now:**

before starting a session I ask: what's the minimum context this task actually needs? then I only load that. if I need more, I add it. it's easier to add than to recover from a bloated session.

token overhead is a monorepo problem more than a model problem. you're not running out of intelligence, you're running out of room for the conversation.

### REDDIT Version

running a monorepo with 50+ AI skills, 17 MCP servers, 3 Next.js sites, and a full GTM pipeline out of one repo. CLAUDE.md has been the difference between Claude actually knowing what to do and Claude hallucinating context from stale memory.

here's what I learned the hard way: a big CLAUDE.md isn't a rule list. it's a routing table.

**what didn't work first**

I started with the obvious stuff. code style rules, no `--no-verify`, always run build checks. fine for a small project. at scale it became noise. Claude would read 300 lines of rules and still open the wrong file because it didn't know what the right file was.

the problem wasn't missing rules. it was missing addresses.

**table-driven context routing**

the most useful thing in my CLAUDE.md is a reference table, not instructions. looks like this:

```
| What | Query |
|------|-------|
| Initiatives backlog | SELECT * FROM initiatives WHERE status NOT IN ('completed','dropped') |
| Active work | SELECT * FROM v_initiatives_active; |
| Evolution state | SELECT * FROM v_dna_snapshot; |
```

instead of trying to explain the whole system in prose, I tell Claude where to look for anything. it runs the query. it knows the state. no stale context.

same pattern for file routing:

```
| Doc | Purpose |
|----------------------------------------|--------------------------------------|
| .claude/teams/TEAM-CONSTRAINTS.md | 9 rules for agent team coordination |
| skills/tier-1-voice-dna/method.md | Recursive Drift methodology |
| docs/ARCHITECTURE.md | System architecture, machine topology|
| gtm-os/CLAUDE.md | GTM operations - tool inventory |
```

Claude reads this at session start and knows exactly where to go for any domain. I don't repeat the architecture in CLAUDE.md. I just tell it where to read it.

**explicit session startup**

this made a bigger difference than I expected:

```
## Session Startup
1. Read .claude/context-handoff.md for previous session state.
2. Read tasks/lessons.md for self-improvement rules . follow every one.
3. Run git status and git log -1 --oneline to know where you are.
4. If working on a partner/client: read their SKILL.md first.
```

most CLAUDE.md files don't have this. Claude doesn't know what order to orient itself unless you write it down. now every session starts with Claude actually knowing where we left off instead of asking me to recap.

**skills system**

the 50 skills live in `.claude/skills/`. each one is a SKILL.md with a trigger condition, tools available, and steps. my CLAUDE.md doesn't document what each skill does. it just says "skills live here, these are the trigger patterns."

actual skill logic is self-contained in the skill file. Claude loads it when triggered. CLAUDE.md stays lean as the system grows.

**self-improvement loop**

this is the one most people skip: `tasks/lessons.md`

every time Claude makes a mistake and I correct it, the rule says update lessons.md with the date and what happened. Claude reads this at session start (step 2 above). mistakes stop recurring because they're written down in a place that gets re-read every session.

not magic. just writing corrections where they can be retrieved.

**what I cut from CLAUDE.md**

- code style (lives in ESLint/Prettier where it belongs)
- tool documentation (moved to `gtm-os/engine/`, one file per tool)
- anything that changes frequently (gets stale, becomes noise fast)
- anything Claude can figure out from reading the codebase

the rule I use: if Claude can derive it from the code, don't put it in CLAUDE.md. CLAUDE.md is for things Claude can't see without being told where to look.

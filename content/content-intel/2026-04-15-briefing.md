## 1. [TUTORIAL] your CLAUDE.md is probably wrong. here's how to actually structure it

**Thesis:** Most builders treat CLAUDE.md like a sticky note. A symlinked, layered CLAUDE.md with scoped instructions per directory is a completely different tool — one that actually changes model behavior at scale.
**Target sub:** r/r/ClaudeCode

### REDDIT Version

your CLAUDE.md is probably wrong. here's how to actually structure it

most people drop one CLAUDE.md in the root, paste some instructions, and call it done. that's a sticky note. it's not a system.

here's what actually changes model behavior at scale: layered CLAUDE.md files, scoped per directory, symlinked from a central source of truth.

**the problem with one root file**

a single CLAUDE.md tries to hold context for your entire repo. so you end up with instructions like "when working on the frontend, do X. when working on the API, do Y. when writing tests, do Z." Claude has to parse all of it every time, and the broader the instructions, the more drift you get in practice.

worse, you can't update it cleanly. you're editing one file that controls everything.

**the actual structure**

Claude reads CLAUDE.md files hierarchically. it loads the root one, then any CLAUDE.md in the current working directory, then any in subdirectories it's touching. they stack.

so the move is:

```
/project-root/CLAUDE.md ← global rules, stack, conventions
/project-root/src/CLAUDE.md ← frontend-specific behavior
/project-root/api/CLAUDE.md ← backend/API rules
/project-root/scripts/CLAUDE.md ← automation, be careful with X
/project-root/tests/CLAUDE.md ← test philosophy, no mocks, etc.
```

each file only contains what's relevant to that directory. when Claude is working in `/api`, it loads the root + `/api/CLAUDE.md`. the `/src` instructions never enter the context. cleaner signal, less noise.

**symlinks for the shared stuff**

I run a monorepo with ~50 skills and multiple projects. there's a core set of instructions that apply everywhere: coding style, how I want diffs formatted, when to ask vs. act, tool preferences.

instead of copy-pasting that block into every repo, I keep one canonical file:

```
~/.claude/shared/core-instructions.md
```

then symlink it into each project:

```bash
ln -s ~/.claude/shared/core-instructions.md ./CLAUDE.md
```

now when I update the core instructions, every project gets it instantly. no drift between repos.

for project-specific additions, I include the symlinked file at the top and extend it:

```markdown
<!-- inherits from ~/.claude/shared/core-instructions.md via symlink -->

## project-specific context

this is a Clay enrichment pipeline. tables are idempotent by design...
```

**what to actually put in scoped files**

root CLAUDE.md:
- tech stack overview
- naming conventions
- how you want decisions surfaced (ask vs. act thresholds)
- what tools are available

directory CLAUDE.md:
- domain-specific behavior
- what patterns to follow or avoid in this part of the codebase
- known gotchas ("the auth middleware runs before X, don't bypass it")
- what files are safe to edit vs. touch with caution

**the thing that actually changed behavior for me**

I added a `scripts/CLAUDE.md` with one rule: never run destructive scripts without showing me the command first and getting explicit confirmation.

Claude was occasionally running cleanup scripts autonomously when asked to "tidy things up." that one scoped instruction killed the behavior entirely. it only applies when Claude is working in that directory, so it doesn't add noise everywhere else.

**tldr**

- one root CLAUDE.md is a sticky note
- layered per-directory files are scoped context
- symlinks keep shared instructions DRY across repos
- the hierarchy stacks cleanly, use it

### REDDIT Version

slash commands in Claude Code are one of the most underutilized features I've seen. people use /commit or /review-pr and stop there. but you can build actual reusable workflow tools that run on demand.

I built one called /extract-design. drop a URL, get back a full design system breakdown: colors, typography, spacing scale, component patterns, and a tailwind config scaffold. here's the exact setup.

**the skill file structure**

Claude Code skills live in `~/.claude/skills/` as markdown files. the filename becomes the slash command. so `extract-design.md` → `/extract-design`.

frontmatter at the top:

```markdown
---
name: extract-design
description: Extract a complete design system from any website URL
---
```

everything after that is your prompt. Claude reads it as instructions and executes.

**the full prompt**

```
You are extracting a complete design system from a website.

The user will provide a URL. Your job:

1. Fetch the full page HTML/CSS using WebFetch
2. Also fetch linked stylesheet paths if discoverable (/css/, /styles/, etc.)
3. Extract and return:

## Colors
- Primary, secondary, accent palette
- Background and surface colors
- Text colors (heading, body, muted)
- Border/divider colors
- Semantic colors (success, error, warning)
- Hex values where possible, CSS variables where not

## Typography
- Font families (heading, body, mono)
- Type scale (sizes used, px and rem)
- Font weights in use
- Line heights
- Letter spacing patterns

## Spacing
- Spacing scale inferred from padding/margin patterns
- Container max-widths
- Section padding patterns

## Shadows & Borders
- Box shadow values
- Border radius patterns
- Border styles

## Component Patterns
- Button variants and states
- Card/surface patterns
- Input styles
- Badge/tag patterns

## Tailwind Config Scaffold
Output a partial tailwind.config.js theme.extend block with extracted values mapped to Tailwind keys.

Be specific. Use real values. If something isn't determinable from the HTML/CSS, say so explicitly . don't guess.
```

**how I actually run it**

open Claude Code, type `/extract-design`, paste a URL. it fetches the page, finds the stylesheet links, pulls those too, and builds the full extraction. 60-90 seconds on a real site. output pastes directly into a design tokens file or tailwind config.

I've used it on competitor sites at the start of new client projects. saves an hour of manual digging in dev tools.

**why the output format matters**

I structured it with the tailwind scaffold as the last section intentionally. the goal isn't just information, it's actionable output. if the last block is always a partial config you can paste somewhere, you skip the "now what do I do with this" step entirely.

not on tailwind? swap that section for whatever your actual next step is. CSS custom properties block, style-dictionary tokens, whatever. the command is only as useful as what it produces.

**variations I've built off this**

➡️ `/extract-design-tokens` -- stripped-down version that only outputs a style-dictionary compatible JSON blob. useful when feeding into a larger pipeline

➡️ `/extract-design-compare` -- takes two URLs and diffs them. built it for a rebrand audit. less stable because two fetches sometimes hit rate limits, but when it works it's clean

also added a second pass where Claude scores brand consistency 1-10 based on how coherent the extracted system looks. surprisingly useful signal for agency work.

**one thing worth knowing**

skills that fetch external URLs depend on WebFetch having access. if you're running in a restricted environment or the site blocks headless requests, it'll fail gracefully and tell you. Firecrawl as an MCP gives you a more reliable fallback for sites with heavy JS rendering.

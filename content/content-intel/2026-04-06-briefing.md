the expensive sessions weren't the complex ones. they were the ones with bad habits baked in.

three things kept showing up:

**1. not checkpointing between milestones**

long sessions compound. if you're 40 messages deep on a debugging rabbit hole and Claude is reading the same 8 files on every turn, you're paying for that history on repeat. I was treating sessions like infinite scratch pads.

fix: `/clear` more aggressively. when you hit a natural milestone, start a new session with a tight handoff prompt:

```
context: fixing a race condition in [file]. confirmed the issue is the mutex lock on line 47. approach is [X]. pick up from here.
```

that 3-sentence handoff costs almost nothing. the 40-message session costs a lot.

**2. MCPs I wasn't actually using**

had 17 MCP servers configured. on any given session, I was using 3 of them. but all 17 were advertising their tool definitions in every session context.

each MCP server injects its schema into your context window every turn. if you have Firecrawl, HubSpot, GitHub, Postgres all loaded... Claude is reading all those tool definitions whether you call them or not.

went through and disabled servers that aren't relevant to the active project. kept only the ones I'd actually invoke in that session. the rest sit in config but don't load.

your MCP list isn't free. prune it per project.

**3. re-reading the same large files over and over**

this one surprised me. I kept asking Claude questions about different parts of the same 1500-line config file throughout a session. each question triggered a re-read.

Claude doesn't cache that across turns. every file read is context.

fix: when you know a file will come up repeatedly, paste the relevant section in your opening message. here's the relevant part of config.ts for this session, then the 40 lines that actually matter. Claude has it in context from the start without re-reading it on every turn.

## 2. [TUTORIAL] DESIGN.md + compiled knowledge graph = Claude Code that doesn't forget who it is across sessions

**problem 1: UI drift between sessions**

you build a component library in session 1. session 3, Claude forgets the button variant names. session 7, it starts inventing spacing patterns that don't match anything. by session 12, your codebase looks like 5 different people built it.

DESIGN.md fixes this.

not a style guide. not a Storybook. a living doc Claude reads first and writes back to. structure I use:

```
# DESIGN.md

## Component contracts
- Button: variant = "primary" | "ghost" | "destructive", size = "sm" | "md" | "lg"
- Card: always uses shadow-sm, never shadow-md
- spacing: 4px base unit, use multiples only (4, 8, 16, 24, 32, 48)

## Color system
- brand-primary: #2563eb
- text-muted: #6b7280
- never use raw hex in components, always use design tokens

## Patterns we've established
- forms use react-hook-form + zod, no exceptions
- modals use the Modal primitive in /components/ui/modal.tsx
- data tables always get loading, empty, and error states

## Decisions log
- 2024-03-15: switched from tailwind arbitrary values to design tokens. reason: Claude kept inventing values
- 2024-03-22: settled on 4px base. reason: 8px felt too loose on dense UIs
```

the decisions log is the part that matters. Claude sees *why* a decision was made, not just what it is. that stops it from helpfully reverting things it doesn't have context for.

in CLAUDE.md, add one line:

```
always read DESIGN.md before touching any UI. always update DESIGN.md after establishing a new pattern.
```

---

**problem 2: token overhead from fat context files**

CLAUDE.md files bloat fast. I've seen repos where CLAUDE.md is 4,000+ tokens just describing the architecture. every session starts with Claude eating the whole file before it does anything useful.

the fix is compiling your knowledge graph.

instead of prose descriptions of how your app works, maintain a `knowledge/` directory with pre-compiled relationship maps:

```
knowledge/
 entities.json # what objects exist and their relationships
 api-map.json # endpoints, what they touch, auth requirements
 decision-log.md # why things are the way they are
 dependency-graph.md # what depends on what
```

```json
 "key_patterns": "always query through getUserById(), never raw prisma.user.findUnique()"

 "Workspace": {
 "fields": ["id", "name", "userId", "settings"],
 "relationships": ["belongs to User", "has many Projects"],
 "primary_file": "lib/models/workspace.ts",
 "key_patterns": "settings is a JSONB blob, use WorkspaceSettings type"
 }
}
```

`api-map.json`:

```json
{

 "GET": "lists workspaces for authed user",

 "touches": ["User", "Workspace"],
 "auth": "required"
 }
}
```

when Claude needs to build a feature that touches Workspaces, you include only `entities.json` and `api-map.json`. ~200 tokens instead of the prose explanation that might run 800+. same signal, fraction of the cost.

CLAUDE.md addition:

```
when working on a feature, read the relevant sections of knowledge/ first.
always update knowledge/ when you add entities, endpoints, or relationships.
```

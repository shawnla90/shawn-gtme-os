what this actually looks like:

- tests pass because the test was written by Claude and it tested the wrong thing

**1. don't ask if it fixed it. ask it to break its own solution.**

most of the debugging time I've burned this year wasn't from bad code. it was from confidently correct-looking code I didn't verify deeply enough.

not a model quality problem. a verification gap.

## 4. [TUTORIAL] CLAUDE.md is not magic — here's why Claude Code ignores yours (and how to fix it)

**Thesis:** The people complaining that Claude stopped following CLAUDE.md rules almost always have the same structural problem: rules without context, constraints without examples, and no enforcement hooks. This is fixable.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=9 voice=9 gap=7 effort=medium

### REDDIT Version

seen this complaint 3x this week in different threads: "Claude keeps ignoring my CLAUDE.md rules"

almost always the same structural problem underneath. here's what's actually happening and how to fix it.

---

**failure mode 1: rules without context**

people write stuff like:

```
- always use TypeScript
- never use var
- write clean code
```

Claude reads this at session start. then 40 messages deep, it stops following it. why?

because "write clean code" isn't a constraint. it's a preference with no behavioral anchor. Claude doesn't know what clean means in your codebase specifically, so it fills in the blank with its defaults.

fix: give the rule a why and a what.

```
## Code Style
- Use TypeScript for all new files. Why: we're migrating from JS incrementally
 and mixing types creates runtime bugs we've already hit.
- Never use var. Use const by default, let only when reassignment is needed.
- "Clean" here means: functions under 30 lines, no nested callbacks,
 explicit return types on all exported functions.
```

now it has something to anchor to.

---

**failure mode 2: no examples**

rules without examples get interpreted loosely. especially for commit messages, naming conventions, file structure.

instead of:

```
- write descriptive commit messages
```

do this:

```
## Commit Messages
Format: <type>(<scope>): <description>

Good: feat(auth): add OAuth2 callback handler
Good: fix(db): handle null response from user query
Bad: fixed stuff
Bad: updates

Types: feat, fix, chore, docs, refactor, test
```

Claude pattern-matches to examples. it's not parsing your rules as hard constraints, it's generalizing from behavioral examples. give it good ones.

---

**failure mode 3: no enforcement hooks**

this is the one most people miss. CLAUDE.md loads at session start, but nothing re-surfaces your rules mid-task.

in a long agentic run (50+ messages, lots of tool calls), your rules slide out of the active context window. Claude isn't ignoring you. the signal just faded.

two fixes:

first, add an enforcement section directly in CLAUDE.md:

```
## Enforcement
Before any file edit: verify you're following the code style rules above.
Before committing: check commit message format.
After writing a new function: confirm explicit return types.
```

sounds obvious but it works. you're giving Claude a self-check loop it can reference.

second, use a UserPromptSubmit hook in your Claude Code settings to inject a reminder on every turn. in `.claude/settings.json`:

```json
{
 "hooks": {
 "UserPromptSubmit": [
 {
 "matcher": "",
 "hooks": [
 {
 "type": "command",
 "command": "echo 'before responding: re-read and follow all rules in CLAUDE.md'"
 }
 ]
 }
 ]
 }
}
```

crude. works. forces re-anchoring on every message instead of only at session start.

---

**the CLAUDE.md structure that actually holds**

after building a lot of these, here's the layout that stays coherent through long sessions:

```
# Project Context
[what this project is, key constraints, non-obvious decisions]

# Code Style
[rules + why + bad/good examples]

# Architecture Decisions
[locked decisions with the reason they're locked]

# Enforcement
[when Claude should self-check before acting]

# Out of Scope
[what not to do or suggest, with why]
```

the out of scope section is underrated. explicit "don't do X because Y" beats "do A, B, C" every time for blocking hallucinated suggestions and unwanted refactors.

---

CLAUDE.md isn't magic. it's a system prompt with no native enforcement loop. treat it like any other prompt: give it context, give it examples, give it a way to re-anchor mid-session.

~/.claude/
 skills/
 core/ # things that work everywhere
 gtm/ # clay, hubspot, instantly patterns 
 dev/ # code review, commits, debugging
 content/ # post drafts, linkedin, reddit
 infra/ # vercel, mcp server management
 MEMORY.md
 projects/
 {project-hash}/
 memory/
```

each skill file gets a front matter block with a description field. that description is what Claude actually uses for discovery when you reference a skill by keyword. if the description sucks, the skill gets ignored.

the pattern that changed everything: skill files that load other skill files. I have a `dev/review.md` that sources `dev/commit.md` when it detects you're about to commit. nested skill composition without nesting the files themselves.

# Skills
Local skills are in ./skills/. Global skills are in ~/.claude/skills/.
Prefer local over global when there's a conflict.
```

that's it. local wins, global is the fallback. no manual path management.

the other thing worth knowing: skill files don't need to be long. my best ones are under 40 lines. the ones that fail are the ones I over-engineered to cover every edge case. a skill that does one thing well beats a skill that does five things poorly.

### REDDIT Version

saw the thread a few weeks back where Karpathy's wiki-as-context pattern was making the rounds here. everyone's applying it to codebases. I took it sideways and applied it to my Clay tables, and it's probably the highest-leverage thing I've done for my Claude Code sessions this year.

background if you missed the pattern: instead of feeding LLMs raw code files, you maintain a set of markdown docs that describe what your code does, why it was built that way, the key patterns, gotchas, etc. Claude reads the wiki, not the repo. token counts drop 80-90%, context quality stays the same or gets better because you're removing noise.

the insight I had: GTM tables are just as context-hungry as codebases. maybe worse.

a Clay table for prospect enrichment might have 60+ columns. formulas that reference other columns. Claygent prompts that assume certain input shapes. HTTP API calls that depend on enrichment order. if you're context-switching between tables or handing off to Claude mid-session, you're either dumping massive exports or hoping Claude guesses right. neither works well.

so here's the structure I use.

**one markdown file per table, stored in `/gtm-wiki/`**

each file has these sections:

```markdown
# [Table Name]

## purpose
one paragraph. what this table does, what decision it drives, what the output is.

## inputs
what data enters this table and from where.
example: "sourced from HubSpot company objects via Zapier trigger on lifecycle stage = MQL"

## key columns
only the columns that other logic depends on or that are confusing:
- `job_title_normalized` . cleaned version of raw HubSpot title. used as input to persona scoring claygent
- `company_size_bucket` . formula: IF(employees < 50, "SMB", IF(employees < 500, "MM", "ENT")). source = Apollo enrichment
- `icp_score` . output of main scoring claygent. 0-100. drives routing logic downstream

## enrichment chain
ordered list of what runs in what sequence and why:
1. apollo enrichment (company data, employees, industry)
2. linkedin scrape via Apify (recent posts, headcount signals)
3. web reveal append (Vector, fires on session-level intent)
4. persona scoring claygent (reads job_title_normalized + seniority + department)
5. ICP score claygent (reads all enrichment outputs, returns 0-100)

## known issues / gotchas
- apollo sometimes returns null for employees on private companies. formula falls back to "Unknown" bucket
- web reveal has ~30% match rate on SMB segment. score claygent handles nulls via IF(ISBLANK) fallback

## dependencies
other tables or systems this feeds:
- output rows with icp_score > 70 export to HubSpot via Zapier
- feeds into email sequence table (see `/gtm-wiki/email-sequencing.md`)
```

that's it. no raw formulas copied in. no full column exports. just the mental model, written once.

**how I use it with Claude Code**

I have a skill that loads the relevant wiki pages at session start based on what I'm working on. if I'm debugging the scoring claygent, it loads the table wiki + the prompt doc for that specific claygent.

for one-off questions I just paste the relevant wiki section manually. takes 10 seconds.

token comparison on my main prospect enrichment table:
- old way: exported column list + formula strings + claygent prompt = ~12,000 tokens
- wiki way: the markdown file above = ~800 tokens

and Claude gives better answers with the wiki because I've already done the editorial work of explaining what matters. raw exports are noisy. the wiki is opinionated.

**the setup I'd recommend if you're starting fresh**

1. pick your most-touched table
2. spend 20 minutes writing the markdown file. don't overthink the structure, just answer: what does this do, what goes in, what comes out, what breaks
3. drop it in a `/gtm-wiki/` folder in whatever directory you use with Claude Code
4. next time you open a Claude session about that table, load the file with Read tool or paste it manually
5. update the wiki when the table structure changes. treat it like comments in code, not documentation

the multiplier effect kicks in when you have 5-10 of these and start cross-referencing them. now Claude can reason about your whole GTM system, not just individual tables in isolation.

one thing I haven't fully solved: keeping the wikis fresh when tables change fast. right now I just update manually after major structural changes. if someone has a better sync pattern I'm genuinely curious.

no gatekeeping. my template + an example wiki file for a lead scoring table are in the comments.

## 2. [TUTORIAL] what actually breaks when you move MCP servers from local to team production (I run 17)

**Thesis:** Running 17 MCP servers locally for months then handing them to a team breaks basically everything — auth, paths, environment assumptions, rate limits. Here's the exact failure list and fixes before your team rage-quits.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=8 voice=9 gap=9 effort=medium

### REDDIT Version

what actually breaks when you move MCP servers from local to team production (I run 17)

spent about 4 months running 17 MCP servers locally before I handed the setup to a small team. assumed it would be a 30-minute handoff. it was a 3-day fire drill. here's every failure point I hit so you don't.

---

**the setup context**

17 servers across clay, hubspot, apify, firecrawl, semrush, github, slack, linear, and a few custom ones I built. all running fine locally on claude desktop via stdio. handed off to 2 other people and basically nothing worked.

---

🔥 **failure 1: absolute paths in claude_desktop_config.json**

my config had stuff like:

```json
"command": "/Users/shawn/projects/mcp-servers/hubspot/index.js"
```

obvious in hindsight. every machine has a different username and project structure. the server just silently fails to load with no useful error.

fix: switch to relative paths using `__dirname` in your server entry, or better, publish your custom servers as npm packages your team installs globally. `npx your-mcp-server` works across machines.

---

🔥 **failure 2: env vars that only exist on your machine**

I had API keys in my shell profile (~/.zshrc) that I'd forgotten I put there 8 months ago. my servers were reading them at runtime with `process.env.HUBSPOT_API_KEY`. works for me, returns undefined for everyone else.

the servers don't crash loudly either. they load fine, then fail on first actual tool call with a cryptic auth error.

fix: require env validation at startup, not at call time:

```js
const apiKey = process.env.HUBSPOT_API_KEY;
if (!apiKey) throw new Error('HUBSPOT_API_KEY is required');
```

this makes it fail loudly at load time so people know immediately what's missing. then use a shared .env template and document every key.

---

🔥 **failure 3: rate limits multiplied by team size**

I built my clay MCP with rate limits that worked fine for one person. 5 requests per second felt safe. two people using it simultaneously blew through api quotas we didn't even know existed.

some APIs (apify, semrush specifically) have daily limits that look fine for solo use and get smoked in a week with 3 people.

fix: add request queuing at the server level before you hand off. also audit every API's actual limits against projected team usage. I used a simple token bucket:

```js
import Bottleneck from 'bottleneck';
const limiter = new Bottleneck({ minTime: 300, maxConcurrent: 2 });
const result = await limiter.schedule(() => apiCall());
```

---

🔥 **failure 4: stdio transport doesn't work for remote or shared setups**

stdio is fine for local claude desktop. the moment you want a server running on a shared machine, or accessed remotely, you need SSE or HTTP transport. I knew this theoretically but didn't think through the implications until someone on the team was on a different machine and couldn't connect.

if your team is all on separate machines running their own claude desktop instances, stdio is actually fine. but if you're running servers on a central machine or trying to share state, you need to rethink the transport layer entirely.

---

🔥 **failure 5: no logging, so nobody can debug**

locally I could just watch the terminal. in a team setup, people would come to me saying "the hubspot server isn't working" with zero other info. took me forever to diagnose because I had no structured logging.

fix: add structured logging from day one:

```js
const log = (level, message, data = {}) => {
 console.error(JSON.stringify({ level, message, timestamp: new Date().toISOString(), ...data }));
};
```

note: log to stderr, not stdout. MCP uses stdout for the protocol. logging to stdout breaks everything.

---

🔥 **failure 6: tool names colliding across servers**

I had two servers that both had a tool called `search`. claude would pick one semi-randomly depending on which loaded first. nobody on the team knew which one was being called.

fix: namespace your tool names. `hubspot_search`, `apify_search`. more typing, way less confusion.

---

**what to do before you hand off**

1. run each server on a clean machine with a fresh env and document what breaks
2. validate all env vars at startup not at call time
3. switch to npx or global npm installs for custom servers
4. add bottleneck or similar for rate limiting
5. namespace every tool name
6. write a one-page runbook: what each server does, what keys it needs, how to restart it

the build phase is the fun part. the handoff phase is where you find out how many assumptions you made.

happy to share my runbook template in the comments if that's useful.

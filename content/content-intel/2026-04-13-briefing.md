56% of my Claude spend was conversations with zero tool use. here's how I audited it

---

ran a cost audit on my Claude usage last week and it was humbling.

more than half my API spend was going to conversation turns that never triggered a single tool call. no file reads, no edits, no bash commands. just me typing context, Claude responding, me refining the prompt, Claude clarifying. back and forth until I actually got to the thing I wanted to build.

I had no idea this was happening because I was only ever looking at total monthly cost, not what the cost was *doing*.

here's how I actually audited it:

**step 1 . pull your usage logs**

if you're using the API directly, you have access to usage data per request. I piped mine into a simple script that tagged each request by whether `tool_use` appeared in the response or `tool_result` appeared in the input.

```python
def classify_turn(message):
 has_tool_use = any(block.get("type") == "tool_use" for block in message.get("content", []))
 has_tool_result = any(block.get("type") == "tool_result" for block in message.get("content", []))
 return "active" if (has_tool_use or has_tool_result) else "overhead"
```

anything that came back as `overhead` was a conversation turn where Claude was just... talking.

**step 2 . calculate token cost per category**

once you have the classification, multiply input + output tokens by your model rate and bucket them. mine looked like:

- active (tool use involved): 44% of spend
- overhead (pure conversation): 56% of spend

that 56% wasn't wasted exactly. some of it was necessary planning or clarification. but a lot of it was me being lazy with my initial prompts and paying to iterate instead of thinking upfront.

**step 3 . identify your overhead patterns**

I went back through the overhead turns and found three recurring patterns:

1. **context re-explaining** . I kept re-establishing project context at the start of sessions because I wasn't using CLAUDE.md files effectively. Claude would ask clarifying questions, I'd answer, it'd ask more. 800 tokens of back-and-forth before a single file got read.

2. **vague task framing** . "help me with the enrichment table" type prompts that required Claude to ask what I actually meant before doing anything. vs "read the Clay HTTP API config in /integrations/clay.ts and tell me why the webhook isn't firing" which goes straight to tool use.

3. **approval loops** . I had flows where Claude would propose a change and wait for my confirmation before writing. legitimate for risky ops, overkill for read-only exploration.

**what I changed:**

- moved all project context into CLAUDE.md so sessions start with orientation already done
- wrote a personal "prompt starter" template that forces me to specify: what file, what the current behavior is, what I want instead
- turned off confirmation prompts for read operations in my permission config

overhead dropped to around 30% the following week. not zero, probably never will be, but the 56% was genuinely recoverable.

the deeper lesson: you can't optimize what you can't see. most people (me included, for months) treat AI spend as one opaque number. breaking it down by *whether anything actually happened* is a different way of looking at it.

### REDDIT Version

how I tier my AI models for GTM workflows: haiku for classify, sonnet for enrich, opus only when it earns it

---

ran the numbers on a Clay table last month. 4,200 rows, every enrichment routed through claude-opus-4-6. the output was great. the bill was embarrassing.

not because opus did anything wrong. because I was using a sledgehammer to sort mail.

here's the three-tier system I use now across Clay, Claygents, and any custom enrichment pipeline:

---

**tier 1 . haiku (classify)**

use this when the question is binary or categorical. haiku is fast, cheap, and doesn't need to reason. it just needs to pattern-match.

examples:
- is this company in a hiring expansion or contraction? (feed it headcount delta + open role count)
- does this LinkedIn post mention a product launch, funding event, or partnership?
- is the job title ICP or not ICP?

in Clay I'll run haiku on a formula column with a simple HTTP API call. prompt is usually under 50 tokens. answer is yes/no or a single category.

cost per 1000 rows: pennies. literally.

---

**tier 2 . sonnet (enrich)**

this is the workhorse. use it when you need synthesis from multiple data points but not strategic reasoning.

examples:
- summarize this company's tech stack based on job descriptions scraped from their careers page
- write a one-sentence pain hypothesis based on: industry, headcount, recent funding, ICP job title
- pull the three most relevant signal sentences from a Firecrawl page pull

sonnet handles most of what I used to route to opus by default. it's fast enough to run in Clay without blowing latency, and the output quality on enrichment tasks is nearly identical.

---

**tier 3 . opus (synthesize)**

opus only fires when the task requires judgment, nuance, or reasoning that sonnet actually fumbles on.

examples:
- writing the first personalized email line based on 6+ enriched signals (intent, trigger, persona, pain hypothesis, company motion)
- generating a multi-touch sequence structure given a buying committee map
- scoring an account's sales-readiness score with weighted reasoning across 8 dimensions

I don't route every outbound touch through opus. just the ones where a generic line will kill the conversion.

---

**the routing logic in practice**

I stack it in Clay like this:

1. haiku column: classify account (ICP tier 1 / 2 / 3 or disqualify)
2. filter: disqualified rows stop here, no further enrichment runs
3. sonnet columns: pain hypothesis, trigger summary, company motion
4. opus column: conditional formula . only runs if haiku classified as tier 1 AND a specific trigger signal is present

that conditional formula is the key. opus cost drops by 60-70% because it only fires on accounts that actually warrant it.

---

**where people waste money**

routing everything to opus because it's the safe default. or worse, not thinking about it at all and just copy-pasting someone else's Clay template that hardcoded the model 8 months ago.

haiku exists for a reason. it's not "worse opus." it's a different tool for a different job.

classify with haiku, enrich with sonnet, synthesize with opus only when the output is load-bearing for revenue.

that's the play.

## 3. [HOT_TAKE] switching to haiku for small tasks to save money is actually making your bill worse

**Thesis:** Smaller models skip prompt caching in ways that blow up token counts on anything with a shared system prompt or context window, meaning your 'cheap' haiku call re-ingests more tokens than a cached sonnet call would have. The math does not work the way everyone assumes.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=9 voice=8 gap=7 effort=low

### REDDIT Version

switching to haiku for small tasks to save money is actually making your bill worse

and I can prove it with math that should be obvious in retrospect but wasn't to me until I audited my usage last week

here's what most people assume: haiku = cheaper per token = lower bill for simple tasks. route the big stuff to sonnet, route the small stuff to haiku, profit.

here's what's actually happening if you have a large shared system prompt or persistent context:

claude's prompt caching works on cache hits. a cached sonnet input token costs $0.30/MTok. an uncached haiku input token costs $0.80/MTok.

so if your system prompt is 50k tokens and it's warming in sonnet's cache, every haiku call you make for a "cheap" subtask re-ingests those 50k tokens at full price. no cache hit. different model, different cache key.

50,000 tokens × $0.80/MTok = $0.04 per haiku call just on the system prompt
50,000 tokens × $0.30/MTok (cached sonnet) = $0.015 per sonnet call

you saved money on the completion side and spent 2.5x more on input. on every single "small" call.

this compounds hard in agentic loops. if you're spinning up haiku to handle tool result parsing or quick classification tasks inside a longer sonnet workflow, you're paying uncached input rates every time because the shared context doesn't transfer between model calls.

the break-even isn't where the per-token price comparison suggests it is. it depends entirely on:
- how warm your cache is on the primary model
- how large your shared context is
- how frequently you're switching models within a single workflow

I found this by pulling my usage logs and noticing my haiku spend was higher than expected relative to call volume. the per-call token count looked fine. the aggregate didn't.

if you're routing between models in a single agentic pipeline, profile the cache hit rate per model before assuming the cheaper model is saving you money. it probably isn't.

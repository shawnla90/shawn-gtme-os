## 5. [CONTRARIAN] CLAUDE.md isn't a rulebook. it's context. and that distinction is wrecking your builds.

**Thesis:** Writing rules in CLAUDE.md doesn't enforce them — it just makes them readable to the model, which can still ignore them. Real workflow governance requires a different layer entirely, and most people haven't figured this out yet.
**Target sub:** r/r/ClaudeCode

### REDDIT Version

most people treat CLAUDE.md like a config file. write the rules in, expect the model to follow them. then they get confused when Claude does whatever it wants anyway.

here's what's actually happening: CLAUDE.md is context injection, not enforcement. the model reads it, weighs it against everything else in the conversation, and decides how relevant it is in the moment. if your task context, prior messages, or even the way you phrased a prompt pushes in a different direction, your "rules" lose.

I've watched this wreck builds in production. team has a solid CLAUDE.md. says things like "never write files without reading them first" or "always check for existing patterns before creating new ones." good intentions. but under task pressure, long context, or when the model gets confident about something, those lines just... don't hold.

the mental model fix is this: CLAUDE.md is a briefing document, not a contract. you're briefing a capable contractor, not configuring a linter.

so if you need actual governance, you need a different layer:

**hooks** . shell commands that fire on tool use events. want to prevent writes to certain paths? put it in a hook, not a markdown file. the hook runs regardless of what the model decided.

**tool permissions** . explicitly scoping what tools are available forces constraints at the execution layer, not the context layer. Claude can't bash rm -rf something if bash is scoped out.

**subagent isolation** . if you're using agents for specific tasks, giving them stripped-down, task-specific context means your rules aren't competing with everything else in the window. smaller context, higher signal, better compliance.

**plan mode as a forcing function** . controversial but: requiring plan approval before execution changes the compliance dynamic. the model is committing to a path before acting, not just receiving instructions.

none of this means CLAUDE.md is useless. it's genuinely valuable for things like tone, workflow patterns, tool preferences, project context. I rely on mine heavily. but I stopped writing enforcement logic in it and started writing orientation logic.

the stuff I actually want enforced lives in hooks and permissions config.

if you're debugging "why does Claude keep ignoring my instructions" . check whether you're using CLAUDE.md for the right job. you probably wrote a briefing doc and expected a guardrail.

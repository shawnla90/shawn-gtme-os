**the audit setup**

Claude Code stores session data at `~/.claude/projects/`. each project folder has conversation files you can inspect. I wrote a quick script to pull token counts per message and started looking for patterns.

you're looking for three things: which messages cost the most, where compaction fired, and how many tool calls happened before Claude got useful output.

**failure mode 1: re-reading cached context**

this one hurt. I kept prompting Claude to "check the current state of X file" mid-session after we'd already read it 20 minutes earlier. the file hadn't changed. Claude would dutifully run Read again. full file cost, zero new information.

fix is simple: before asking Claude to read something, check if it came up earlier in the session. if it did, reference it directly. "the UserService file we looked at earlier" costs almost nothing vs. triggering another Read.

**failure mode 2: lazy search prompts**

I'd say things like "find where we handle auth" instead of "check src/middleware/auth.ts". Claude would run Glob, then Grep, then maybe another Grep with a refined pattern, then Read the file.

four tool calls to get to something I already knew the path of.

the fix: when you know the file, say the file. when you genuinely don't know, give Claude enough constraint that it finds it in one pass. "search for the auth handler in src/middleware" beats "find where we handle auth" by 3-4 tool calls on average.

**failure mode 3: bad compaction timing**

compaction is supposed to help but it will absolutely wreck you if it fires mid-implementation. I had sessions where compaction hit right after I'd set up a complex context (here's the bug, here's the relevant files, here's the constraint), and then Claude continued with a compressed version that missed key details. I'd re-explain. more tokens.

what I do now: when I'm about to start a dense implementation chunk, I check context usage first. if I'm past 60-70%, I either finish the current thing and start a new session with clean context, or I front-load a summary that will survive compaction well.

you can see context usage in the bottom right of the Claude Code interface. I check it before kicking off anything that needs sustained focus.

**failure mode 4: CLAUDE.md not pulling its weight**

every session without a good CLAUDE.md means you're either re-explaining your stack manually or Claude is making wrong assumptions and you're correcting them. both cost tokens.

my CLAUDE.md now has: tech stack, file structure, key conventions, common commands, and the things Claude gets wrong without being told. that last category is the most valuable. "we use zod for validation, not yup" is one line that has saved me probably 20 correction loops.

**running the actual audit**

1. pick 3-5 recent sessions that felt expensive or frustrating
2. look at the session file and identify the 5 most token-heavy exchanges
3. for each one, ask: was this tool call necessary? did I already have this context? could I have been more specific in the prompt?
4. look for patterns. your failure modes will cluster.

mine were almost entirely in the search/read category. some people's will be compaction. some will be prompt verbosity. the audit tells you which lever to pull.

once I fixed my prompting habits and started managing compaction intentionally, my context window stopped feeling like a problem. the same sessions I used to hit limits on now finish with room to spare.

the context window isn't the issue for most of us. the issue is 40% of what we're putting in it.

something I noticed after 3 months of building agentic GTM pipelines: Claude will skip your steps if you let it.

not because it's broken. it's optimizing for what it thinks you want, and when your task is ambiguous, it makes assumptions. in a content enrichment pipeline, that assumption is usually something like: I'll skip the verification step because the data looks fine.

it's not fine. the downstream breakage is always the thing you didn't check.

CLAUDE.md fixed this. here's the exact pattern I use.

---

**the core problem**

most CLAUDE.md files look like this:

```
# Project
This is a GTM enrichment project.
Use TypeScript. Follow existing patterns.
```

fine for context. does nothing for behavior.

what actually drives step discipline is combining three things:
1. explicit sequential contracts
2. a forbidden shortcuts list
3. checkpoint verification format

---

**sequential contracts**

instead of describing what the pipeline does, describe what Claude is NOT allowed to skip:

```markdown
# Pipeline Step Discipline

Steps execute in order. Each step must produce a verified output before the next step begins.

NEVER proceed to the next step if:
- The previous step returned an error or empty result
- The previous step output hasn't been logged to the task file
- You haven't confirmed the expected output format matches the actual output

If a step fails, STOP and report the failure. Do not attempt to infer what the output would have been.
```

the key phrase is "stop and report." left to its own judgment, Claude will often try to recover silently. this makes failure loud.

---

**forbidden shortcuts list**

this is the one most people skip entirely. I keep a running list of shortcuts Claude has actually taken in my builds:

```markdown
# Forbidden Shortcuts

Never do these, even if you think the result will be the same:
- Do not skip the deduplication check because "the data looks clean"
- Do not infer a company's HubSpot ID without running the lookup
- Do not assume a Clay enrichment returned valid data without checking null fields
- Do not combine step 3 and step 4 into a single operation to save time
- Do not write to HubSpot before the enrichment validation step completes
```

every line here came from a real incident. the deduplication one happened twice before I wrote it down.

---

**checkpoint format**

I define a mandatory output format for each completed step:

```markdown
# Step Completion Format

After completing each step, output:

STEP [n] COMPLETE
Input: [what you received]
Output: [what you produced]
Issues: [any anomalies, even minor ones]
Next: [what step n+1 expects from this output]
```

sounds bureaucratic. it's load-bearing.

it forces Claude to reason about the handoff between steps, not just execute the step in isolation. the "Next" field specifically catches most of the silent assumption bugs because it requires Claude to think forward, not just backward.

---

**putting it together**

my full CLAUDE.md for a GTM enrichment pipeline is around 180 lines. roughly 60% of that is behavioral contracts like these. the other 40% is project context, tool patterns, and a section called "known failure modes" where I document past incidents.

the thing that changed when I started treating CLAUDE.md as a behavioral spec instead of a project readme: I stopped finding broken outputs halfway through a 500-row enrichment run.

that's the actual ROI. not cleaner code. fewer Monday morning fires.

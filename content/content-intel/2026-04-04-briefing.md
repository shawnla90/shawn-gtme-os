```

- /lib/env.ts → typed env var access. getEnv(key, required?)
```

**why this works**

Claude doesn't need to read a file to know what it does if you tell it. when I ask Claude to update the clay-enrichment skill to handle rate limits, it goes directly to `/skills/clay-enrichment/index.ts` and `fetchWithRetry()` in `/lib/fetch.ts` without any exploration detour.

## 2. [TUTORIAL] how I run 4 Claude Code sessions in parallel on the same codebase without conflicts

**Thesis:** Git worktrees let you parallelize completely independent workstreams in Claude Code. For GTM automation builds, this means running enrichment logic, email sequencing, and webhook handlers simultaneously instead of waiting in serial.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=8 voice=9 gap=6 effort=medium

### REDDIT Version

how I run 4 Claude Code sessions in parallel on the same codebase without conflicts

been doing this for a few months and it's genuinely changed how fast I ship. the trick is git worktrees. most people don't know this exists.

**the problem first**

I'm building GTM automation. one codebase handles enrichment logic, email sequencing, webhook handlers, and CRM sync. these are all independent. but if I open 4 Claude Code sessions on the same repo, they're all touching the same working tree. conflict city.

the naive fix is 4 separate clones. that works but it's disgusting. you end up with stale code everywhere and merging becomes a nightmare.

**git worktrees are the actual answer**

a worktree lets you check out multiple branches simultaneously, each in their own directory, all pointing to the same git object store. no duplicating the repo. no stale code. just isolated working directories.

setup looks like this:

```bash
# from your main repo directory
git worktree add ../project-enrichment feature/enrichment-logic
git worktree add ../project-email feature/email-sequencing 
git worktree add ../project-webhooks feature/webhook-handlers
```

now you have:
- `~/project` → main branch (or whatever you're on)
- `~/project-enrichment` → feature/enrichment-logic
- `~/project-email` → feature/email-sequencing
- `~/project-webhooks` → feature/webhook-handlers

open Claude Code in each directory as a separate session. they're completely isolated. claude in one worktree has no idea the others exist.

**what this looks like in practice**

I'll kick off a build and have:

- session 1 in `~/project-enrichment` working on the Clay HTTP API integration
- session 2 in `~/project-email` building the Instantly sequence trigger logic
- session 3 in `~/project-webhooks` handling inbound webhook parsing from HubSpot
- session 4 in `~/project` (main) reviewing what's coming in or handling a hotfix

they all run against the same git history so shared utilities, types, and config are consistent. but none of them can stomp each other.

**a few things that bit me early**

if your project has a `.env` file or any local config that isn't tracked by git, you need to copy it into each worktree manually. the worktree only shares what's in the object store.

also watch your `node_modules`. if you're in a monorepo or have a build step that writes to the project root, you'll want to run `npm install` (or whatever) in each worktree separately. they don't share node_modules by default.

and name your worktree branches something meaningful. `feature/worktree-1` will haunt you during merge review.

**cleanup**

when you're done:

```bash
git worktree remove ../project-enrichment
git worktree remove ../project-email
git worktree remove ../project-webhooks
```

then merge your branches normally. since all four sessions were working on isolated branches from the same base, merges are usually clean unless you had overlapping files (which defeats the purpose of parallelizing in the first place).

**the real unlock**

the reason this matters for agentic work specifically: Claude Code sessions are long. when you're doing a real build, not just a quick edit, a session can run 20-30 minutes of back-and-forth. waiting in serial means your 4-feature sprint takes 2 hours instead of 30 minutes.

this also pairs well with CLAUDE.md files per worktree if you want claude scoped to just that feature's context without loading the whole project.

been using this pattern on my GTM automation monorepo and it's the main reason I can ship 3-4 independent features in a single work session.

treating Claude like a creative director is still the wrong frame. treat it like a junior SDR.

the creative director framing is everywhere right now. collaborate with it, give it creative latitude, let it surprise you. and for certain use cases that works fine.

but if you're building GTM workflows, content pipelines, or anything that needs to run consistently at scale, that framing will burn you.

here's the difference in practice:

a creative director gets a brief and interprets it. that's the job. you want variance, you want perspective, you want them pushing back on the brief itself.

a junior SDR gets a playbook. specific ICP criteria, a sequence, call dispositions, objection scripts. they follow it exactly until they have enough reps to know when to deviate. if they go off-script too early, quality crumbles.

Claude, left to interpret, will drift. it'll write something that sounds good and misses the point. it'll optimize for the output feeling right rather than the output being right.

what actually fixes this:

**ICP definition in the prompt.** not "write for GTM engineers," but "write for ex-SDR turned GTM engineer, 2-5 years experience, runs Clay or similar tools, skeptical of hype, values specificity over polish." that's an ICP, not a vibe.

**the playbook = structured context.** a CLAUDE.md per project with tone rules, anti-patterns, series format, specific examples of good vs. bad output. not a style guide, a rep brief. Claude reads it at the start of every session.

**feedback loops, not regeneration.** when output is wrong, most people just regenerate. that's vibes-based iteration. instead, diagnose what rule broke and update the playbook. the next 100 outputs benefit, not just the next one.

**call dispositions.** in SDR terms: labeling why a call failed. in Claude terms: "too generic," "missed the practitioner angle," "structured as listicle instead of narrative," "passive voice throughout." specific failure modes go into the playbook so they stop recurring.

the creative director framing gives Claude room to interpret. fine when you want one great output. falls apart when you need 40 consistent ones.

the junior SDR framing constrains the interpretation surface. the outputs get boring in the best way. boring means reliable. reliable means you can build on it.

most people are still debugging prompts one at a time. the unlock is treating your prompt system like a process you manage, not a message you perfect.

### REDDIT Version

30x less context per task: how to run a local LLM as a Claude Code subagent

most people using Claude Code are burning Opus/Sonnet tokens on tasks that don't need it. file classification, log parsing, regex generation, simple summarization. the model doesn't matter for those. throughput matters.

here's the pattern: Ollama running locally, exposed as a bash tool, letting Claude delegate cheap subtasks to Qwen3:6b instead of spinning up full context windows on the primary model.

---

**the problem**

every Agent tool call in Claude Code spins up a new context window. if you're running subagents on 40 files to do something repetitive, that's 40 context windows burning API quota. it hits rate limits fast, adds latency, and eats into the context budget you actually need for reasoning.

---

**the setup (~15 min)**

step 1: get Ollama running

```bash
brew install ollama
ollama pull qwen3:6b
ollama serve
```

runs an OpenAI-compatible API at `localhost:11434`

step 2: create a local LLM wrapper script

```bash
#!/bin/bash
# local-llm.sh
PROMPT="$1"
curl -s http://localhost:11434/api/chat \
 -H "Content-Type: application/json" \
 -d "{
 \"model\": \"qwen3:6b\",
 \"messages\": [{\"role\": \"user\", \"content\": \"$PROMPT\"}],
 \"stream\": false
 }" | jq -r '.message.content'
```

step 3: register it as a tool in CLAUDE.md

```markdown
## Tools

- local_llm: runs a prompt against local Qwen3 via Ollama.
 use for: file classification, simple transforms, regex generation,
 log summarization, boilerplate generation.
 do NOT use for: multi-step reasoning, tasks requiring cross-file context.
 command: bash ~/scripts/local-llm.sh "$PROMPT"
```

now Claude routes cheap tasks there automatically. you can also be explicit in prompts: "classify these files using local_llm, then bring the results back for the main pass."

---

**what I route to it**

- classifying 200+ files before the main agent processes them
- summarizing raw API responses before Claude reads them 
- generating boilerplate (test stubs, variable names, repetitive config blocks)
- parsing logs where the pattern is obvious and consistent

rule of thumb: if the task is self-contained and the output format is predictable, local model. if it needs context from more than 2 sources or requires actual reasoning, stay on Claude.

---

**the math**

primary model subagent call: ~6-10k tokens per invocation on your API quota 
local model call: 0

on a workflow with 30 subtasks, that's potentially 200k+ tokens you're not burning. on M-series chips Qwen3:6b runs around 40 tok/sec with no rate limits.

---

**the one gotcha**

if you route reasoning-heavy tasks to the local model, you'll get garbage back and your main agent has to redo the work anyway. that's worse than just keeping it on Claude. be specific in CLAUDE.md about what the local tool is and isn't for. Claude respects those constraints well if you write them clearly.

here's what changed when I started treating it that way: I built an eval loop (I call it GEPA . Grounded Eval and Prompt Analysis) that scores CLAUDE.md outputs the same way I'd score a customer-facing prompt.

the setup:

1. define test cases. real tasks the agent should handle: write a commit message, refactor this function, add a new tool. 20-30 cases minimum.
2. score outputs on a rubric. I use Haiku 4.5 as the grader . fast, cheap, consistent. rubric: did it follow the style guide? did it use the right tool? did it skip the thing I said not to do?
3. run the eval. log pass rate.
4. edit CLAUDE.md. run again.
5. repeat until improvement plateaus.

baseline: 65% pass rate.
after 4 iterations of CLAUDE.md edits: 85%.

no model change. no code change. just prompt engineering applied to the file that runs your agent.

specific changes that moved the needle:

**negative constraints outperform positive instructions.** "don't add docstrings to functions you didn't touch" beats "only modify what's necessary." the model anchors on the verb harder than the framing.

**specificity over principles.** "use Edit over Bash for file modifications" lands better than "prefer dedicated tools when available." name the tools. name the anti-pattern.

**order matters more than you'd expect.** I moved my highest-impact rules to the top 30 lines. context window attention isn't uniform . early tokens carry more weight.

**inline examples outperform abstract rules.** two-line concrete examples inside CLAUDE.md are more sticky than paragraphs about intent. "do this: `git commit -m ...` not this: `echo ... > file`" is immediately actionable.

the eval runner is about 80 lines of Python using the Anthropic SDK. it loads CLAUDE.md, constructs a synthetic session, runs each test case, grades with Haiku 4.5, logs to JSON. full run takes about 4 minutes.

happy to drop the script in the comments if useful. not pretty, but it works.

the point isn't the specific numbers. the point is that vibes-based CLAUDE.md editing gives you no signal. once you have a pass rate you have a feedback loop. once you have a feedback loop, you can ship the version that actually scores better.

treat it like a prompt. run evals. iterate.

quick background for anyone who hasn't seen it: GEPA (Generative Evaluation and Prompt Adaptation) is an open-source framework that treats any prompt file like a tunable parameter. instead of manually tweaking your CLAUDE.md and hoping things get better, it runs structured eval loops, scores agent outputs against defined criteria, and proposes targeted edits to the instruction file. then it re-runs, scores again, iterates.

the insight that made this click for me: your CLAUDE.md is just a prompt. and prompts have measurable quality. so why aren't we measuring it?

**how the loop actually works**

1. you define an eval suite . a set of tasks your agent should handle correctly
2. GEPA runs your agent against the suite with the current CLAUDE.md
3. it scores each run (pass/fail or rubric-based depending on your config)
4. it reads the failures, proposes specific line-level edits to CLAUDE.md
5. re-runs the eval suite with the updated file
6. repeats for N iterations or until pass rate plateaus

the output is a diff of your CLAUDE.md with a per-iteration pass rate log. you review it, merge what you want, throw out what you don't.

**what I actually eval against**

this is where most people get stuck. the evals have to be grounded or the optimization drifts toward gaming the metric.

my current suite has three categories:

- **task completion** . agent gets a realistic sub-task (write a bash script that does X, add a function to this file, debug this error). scores on whether the output is correct and complete without extra prompting
- **constraint following** . tasks that should trigger specific behaviors from CLAUDE.md (don't use subprocess, always use absolute paths, never create new files unless asked). scores on whether the constraint was respected
- **multi-step coherence** . two or three chained tasks where the agent needs to hold context correctly across steps. this is the one that breaks most often and surfaced the most useful CLAUDE.md improvements

**what actually changed in my CLAUDE.md after 6 iterations**

some of the proposed edits were noise. but the ones that stuck:

- added explicit task bracketing instructions ("always confirm scope before writing code, one sentence max")
- tightened the tool priority hierarchy (GEPA caught that I had conflicting instructions between two sections I'd written months apart)
- added a negative example block for a pattern the agent kept falling into (over-using Bash when Glob would work)
- rewrote the file creation rule from vague to specific with an example of when to and when not to

pass rate went from 61% to 84% across my eval suite. not scientific, but directional.

**the config that matters most**

temperature on the proposal step. too high and it suggests wild structural rewrites. too low and it just rewrites sentences without changing behavior. I landed on 0.4 for the edit proposals after a lot of trial and error.

also: keep your eval tasks short and single-responsibility. if a task is ambiguous, GEPA will optimize toward whatever interpretation scores best, not necessarily the one you wanted.

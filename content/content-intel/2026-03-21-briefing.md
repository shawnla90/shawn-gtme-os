**Thesis:** Claude Code has no persistent memory by default, so every session starts cold. A timestamped handoff file system solves this - here's the exact setup I've been running in production across parallel terminals.
**Target sub:** r/r/ClaudeCode

### REDDIT Version

how I got Claude Code to never forget my project between sessions (my full context handoff system)

every session starts cold by default. claude doesn't remember what you were building, what broke last night, what decisions you made. if you're running multiple terminals or parallel agents, it's even worse . they stomp on each other.

I've been running a timestamped handoff file system in production for a few weeks now. here's exactly how it works.

**the core idea**

instead of one context file that gets overwritten, each session writes its own handoff to `~/.claude/handoffs/` with a timestamp slug:

```
~/.claude/handoffs/2026-03-22_143022_niobot-sse-fix.md
~/.claude/handoffs/2026-03-22_151847_reddit-pipeline.md
```

parallel sessions never collide. each terminal writes its own file. no more "wait did I just overwrite yesterday's context?"

**on session end**

I have this in my CLAUDE.md so claude auto-fires it without me asking:

```
On Session End: write handoff to ~/.claude/handoffs/YYYY-MM-DD_HHMMSS_<slug>.md
Never overwrite another session's handoff.
Do NOT ask permission - just write it.
```

the slug is a 2-3 word summary of what was worked on. makes it scannable.

**on session start**

```
1. read all unconsumed handoffs: ls ~/.claude/handoffs/*.md 2>/dev/null | grep -v '_done.md$'
2. check legacy location: ~/.claude/context-handoff.md
3. after reading, mark each consumed: rename file.md → file_done.md
4. clean up old consumed handoffs: find ~/.claude/handoffs -name '*_done.md' -mtime +7 -delete
```

the `_done.md` rename is key. it's the consumed/unconsumed state without any database or registry. just the filesystem.

**the handoff template**

the actual file content matters more than the filename. mine includes:

- what was completed this session
- what's in progress / next steps
- any blockers or decisions made
- relevant file paths and context
- what NOT to re-explore (saves 10 mins of redundant reads)

the "what NOT to re-explore" section alone cut my session startup time in half. claude won't re-read 8 files it already knows the answer to.

**parallel agent case**

I run 3-4 terminals sometimes. each writes its own timestamped file. on the next main session, claude reads all unconsumed ones in order, gets full picture from every parallel thread, marks them all done.

before this, parallel agents would either: a) all try to write to the same file and race condition, or b) I'd have to manually consolidate context. now it just works.

**the CLAUDE.md hook**

this only works if claude actually does it. I added this to my project CLAUDE.md:

```
## Context Handoff System
Handoffs are parallel-safe. Multiple terminals can write handoffs without overwriting each other.

On Session Start:
1. Read all unconsumed handoffs: ls ~/.claude/handoffs/*.md | grep -v '_done.md$'
2. After reading, mark each consumed: rename file.md to file_done.md
3. Clean up old: find ~/.claude/handoffs -name '*_done.md' -mtime +7 -delete

On Session End:
Write handoff to ~/.claude/handoffs/YYYY-MM-DD_HHMMSS_<slug>.md
Do NOT ask permission - just write it.
```

the "do not ask permission" is important. otherwise claude will ask "should I write a handoff?" at the end of every session and you'll say no because you're tired. just make it automatic.

**cost**

handoff files are tiny. mine average ~800 tokens. at the start of a session, reading 2-3 of them costs maybe 3k tokens. that's nothing compared to re-exploring the codebase from scratch every time.

## 2. [TUTORIAL] MCP is costing you 37% more tokens than it should - here's how to audit and fix it

**Thesis:** Most people stack MCP servers without thinking about schema bloat. Every tool definition loads on every call. Here's how to profile your MCP footprint, apply lazy loading, and trim schemas to cut real costs at scale.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=7 voice=8 gap=7 effort=medium

### REDDIT Version

most people treat MCP like a plugin store. install everything, never look back.

but every tool definition you load gets injected into the context window on every single API call. not just when you use the tool. every call.

I audited my setup last week. 17 MCP servers, average 8 tools each = 136 tool schemas loading constantly. ran it through a tokenizer. **4,200 tokens per call just in tool definitions.** at Sonnet 4.6 pricing that's not nothing, especially at scale with automated pipelines.

here's how to profile and fix it.

---

**step 1: count your actual MCP token footprint**

grab your `claude_desktop_config.json` and list every server. then for each server, check how many tools it exposes:

```bash
# for servers with a CLI or inspector
npx @modelcontextprotocol/inspector --server "your-server-command"
```

or just log the tool definitions directly. the format is JSON schema . name + description + inputSchema. paste the whole block into a tokenizer (tiktokenizer.vercel.app works fine). you want the raw number before you optimize.

my breakdown was embarrassing: posthog MCP alone was 1,100 tokens. attio was 900. and I wasn't even using most of those tools in the session.

---

**step 2: apply lazy loading via config profiles**

Claude Code doesn't have native lazy loading but you can fake it with multiple config files:

```bash
# ~/.claude/mcp-profiles/gtm.json . only CRM + enrichment tools
# ~/.claude/mcp-profiles/dev.json . only code + browser tools 
# ~/.claude/mcp-profiles/content.json . only publishing tools
```

then symlink whichever profile you need:

```bash
cp ~/.claude/mcp-profiles/gtm.json ~/.claude/mcp.json
```

crude but it works. my GTM sessions dropped from 136 tools to 31 tools loaded. that's the majority of the bloat gone before touching a single schema.

---

**step 3: trim schemas without breaking functionality**

most MCP server descriptions are written for humans, not for token efficiency. the model doesn't need a paragraph explaining what "create a record" does.

before:
```json
{
 "name": "create_record",
 "description": "Creates a new record in the specified object type within the Attio CRM system. This tool allows you to add new companies, people, deals, or other custom objects to your workspace with their associated attributes and values.",
 "inputSchema": { ... }
}
```

after:
```json
{
 "name": "create_record", 
 "description": "Create a new CRM record (company/person/deal).",
 "inputSchema": { ... }
}
```

same behavior. ~60 tokens saved per tool. across 136 tools that's real money.

for self-hosted or open source MCP servers you can edit the source directly. for closed servers you can write a thin wrapper that intercepts the tool list response and strips descriptions before passing to the model.

---

**step 4: audit parameter schemas for redundancy**

this is where it gets tedious but worthwhile. a lot of MCP servers include `default`, `examples`, and `deprecated` fields in their JSON schemas. the model ignores most of these. strip them:

```python
def trim_schema(schema):
 remove_keys = {'examples', 'default', 'deprecated', '$comment', 'title'}
 if isinstance(schema, dict):
 return {k: trim_schema(v) for k, v in schema.items() if k not in remove_keys}
 if isinstance(schema, list):
 return [trim_schema(i) for i in schema]
 return schema
```

run this on the inputSchema block of each tool definition. I got another 8-12% off after the description trimming.

---

**the result**

before: ~4,200 tokens per call baseline
after profiles + trimming: ~1,100 tokens per call

on a pipeline running 500 calls/day that's 1.55M tokens/day in savings. at Sonnet 4.6 rates that's $11-12/day back. compresses fast.

the 37% number in the title is actually conservative . I've seen setups with 25+ MCP servers where the footprint is 40-50% of total context before you write a single word.

worth auditing before you add your next MCP server.

## 7. [PSA] 5 security holes that keep showing up in vibe-coded apps before people ship them

**Thesis:** AI-assisted builds are fast but the shortcuts are consistent and predictable. The same five blind spots appear in almost every Claude Code project I've reviewed - here's the checklist that catches them before they hit prod.

five security holes that keep showing up in vibe-coded apps before people ship them

not blaming the AI. these are predictable shortcuts that happen when you're shipping fast and the model is optimizing for "works" not "secure." here's the checklist I run before anything goes anywhere near prod:

---

**1. API keys leaking into client-side code**

claude scaffolds fast. sometimes too fast. check your next.js components for anything that looks like:

```
const apiKey = process.env.NEXT_PUBLIC_OPENAI_KEY
```

`NEXT_PUBLIC_` prefix means it ships to the browser. every key that needs to stay secret should live in a server route, not a component. grep your whole repo for `NEXT_PUBLIC_` and audit every hit.

**2. unauthenticated API routes**

the model builds the route, makes it work, moves on. authentication gets added "later." later never comes.

check every `/api/` route you have. ask: what happens if someone hits this without a session? I've seen admin endpoints, data deletion routes, and payment triggers all sitting open. middleware-level auth is the fix - don't do it per-route.

**3. no rate limiting anywhere**

claude code builds functional endpoints. it doesn't think about abuse. if you have a form submission, an AI inference call, or anything that touches an external API... it's probably wide open to hammering.

`upstash/ratelimit` with redis takes about 10 minutes to add. do it before launch not after your openai bill spikes.

**4. SQL injection via raw query strings**

this one shows up in projects that used the HTTP API or custom db integrations. parameterized queries exist for a reason. if you see anything like:

```
db.query(`SELECT * FROM users WHERE email = '${email}'`)
```

that's a problem. use prepared statements or your ORM's query builder. always.

**5. missing input validation on user-supplied data**

claude builds optimistic code. it assumes the input is what you expect. zod or yup on every form and API handler, server-side. client-side validation is UX. server-side validation is security.

## 8. [RESOURCE_SHARE] how I run 3-4 Claude Code sessions in parallel without them stepping on each other - full setup

**Thesis:** Parallel Claude Code agents across git worktrees sounds chaotic but there's a system: tmux layout, isolated worktree per feature, and a handoff protocol between sessions. Here's exactly how it's configured.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=8 voice=9 gap=8 effort=high

### REDDIT Version

running 3-4 Claude Code sessions in parallel without them stepping on each other - full setup

been doing this for about 6 weeks. the first week was a disaster. sessions fighting over the same files, one agent reverting another's work mid-build, git state becoming unreadable. here's what fixed it.

---

**the core problem**

when two agents edit the same files, you get merge conflicts. when they both run builds, they trash each other's compiled output. when one agent commits mid-flow, it corrupts the other agent's context. you need physical isolation, not just "try to stay in your lane."

---

**git worktrees are the answer**

`git worktree` lets you check out multiple branches simultaneously in separate directories - all sharing the same .git history.

```bash
git worktree add ../myrepo-feature-a feature-a
git worktree add ../myrepo-feature-b feature-b
git worktree add ../myrepo-hotfix hotfix/critical-bug
```

now each session gets its own directory. agent in `myrepo-feature-a` physically cannot touch files in `myrepo-feature-b`. builds are isolated. git states are isolated.

---

**tmux layout**

4-pane setup, one pane per agent:

```bash
tmux new-session -d -s parallel -x 220 -y 50

tmux split-window -h
tmux select-pane -t 0
tmux split-window -v
tmux select-pane -t 2
tmux split-window -v

tmux send-keys -t parallel:0.0 'cd ../myrepo-main && claude' Enter
tmux send-keys -t parallel:0.1 'cd ../myrepo-feature-a && claude' Enter
tmux send-keys -t parallel:0.2 'cd ../myrepo-feature-b && claude' Enter
tmux send-keys -t parallel:0.3 'cd ../myrepo-hotfix && claude' Enter
```

I keep main top-left as the integration point. the three parallel agents live in the other panes. I can see all four simultaneously without switching windows.

---

**the handoff protocol**

this is the part that makes parallel actually usable long-term.

each agent writes a handoff file when it finishes or pauses:

```
~/.claude/handoffs/YYYY-MM-DD_HHMMSS_<slug>.md
```

timestamped filenames mean parallel sessions never overwrite each other. format I use:

```markdown
# Handoff: feature-a / 2026-03-22 14:32

## What's done
- built the auth middleware
- all tests passing

## What's next
- integrate with the new user model (PR #142 just merged to main)
- rebase against main before merging

## Known issues
- rate limiter config is hardcoded, needs env var
```

when I start a new session or context gets long, the agent reads all unconsumed handoffs first, then renames them to `*_done.md`. no context is lost across sessions.

I add these instructions to my CLAUDE.md so every session inherits them automatically.

---

**branch hygiene rules**

each worktree lives on its own branch. never reuse a branch across worktrees - the shared .git gets confused. naming convention I use:

```
feature/[ticket-id]-[short-desc]
hotfix/[bug-name]
```

I keep a plain text file tracking which worktree maps to which branch:

```
worktrees.txt:
../myrepo-feature-a → feature/241-auth-refactor
../myrepo-feature-b → feature/244-new-dashboard
../myrepo-hotfix → hotfix/login-redirect-bug
```

---

**gotchas to watch for**

shared build caches still collide. if you're running `next build` or `tsc` in parallel across worktrees, they'll fight for `.next/` or `dist/`. either use separate output dirs per worktree or stagger the builds.

node_modules is shared by default. if one agent runs `npm install` mid-session, it affects all worktrees. pre-install everything before starting the parallel sessions.

if your CLAUDE.md references absolute paths, each worktree inherits the same file. usually fine, but audit for any assumptions about being in the root repo.

---

**actual output delta**

running parallel vs. sequential is roughly 3x feature throughput per day for me. the mental model shift is real: instead of waiting for one agent to finish, you brief 3-4 simultaneously on independent work streams, go do something else, then review all outputs together. it starts to feel more like engineering manager behavior than individual developer behavior.

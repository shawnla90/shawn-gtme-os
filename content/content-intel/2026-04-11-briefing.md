### REDDIT Version

most people on the $200/month plan don't find out they have a token problem until Claude starts refusing completions mid-session. by then it's too late to course-correct, and you're just waiting for the reset.

I burned through a bunch of sessions before I figured out the issue wasn't how much I was using Claude. it was that I had no idea *which workflows* were doing the burning.

here's what I built to fix it.

---

**the core problem**

Claude Code doesn't give you per-workflow token breakdowns natively. you get a vague sense of "context used" but nothing that tells you "that CLAUDE.md you're loading on every session is costing you 8k tokens before you type a single thing."

so I started logging it manually.

---

**the setup**

create a file at the root of your project: `.token-log.jsonl`

add this to your `.gitignore` immediately:
```
.token-log.jsonl
```

then create a small logging hook. I use a shell alias that wraps certain Claude Code session starts:

```bash
claude_tracked() {
 local workflow_name="$1"
 shift
 local start_time=$(date -u +%s)
 
 # log session start
 echo "{\"event\":\"start\",\"workflow\":\"$workflow_name\",\"ts\":$start_time}" >> .token-log.jsonl
 
 # run claude with remaining args
 claude "$@"
 
 local end_time=$(date -u +%s)
 echo "{\"event\":\"end\",\"workflow\":\"$workflow_name\",\"ts\":$end_time,\"duration\":$((end_time - start_time))}" >> .token-log.jsonl
}
```

then call it like:
```bash
claude_tracked "pr-review" --print "review this PR for logic errors"
claude_tracked "test-gen" --print "write tests for auth module"
```

---

**the CLAUDE.md audit**

before anything else, run this:

```bash
wc -c CLAUDE.md
```

then multiply roughly by 0.25 to get an approximate token count. if your CLAUDE.md is 40kb, that's ~10k tokens loading on every single session. that adds up fast when you're running 20+ sessions a day.

I trimmed mine from 28k characters to 11k characters by pulling out stuff that was rarely relevant and moving it to separate context files I only load when needed:

```bash
# instead of one giant CLAUDE.md
# use targeted loads
claude --context ./contexts/testing.md "write tests for..."
claude --context ./contexts/deploy.md "help me debug this deployment..."
```

---

**the visibility layer**

after a week of logging, I built a quick analyzer in Python:

```python
import json
from collections import defaultdict
from pathlib import Path

log_path = Path(".token-log.jsonl")
sessions = defaultdict(list)

with open(log_path) as f:
 for line in f:
 entry = json.loads(line)
 sessions[entry["workflow"]].append(entry)

for workflow, events in sessions.items():
 starts = [e for e in events if e["event"] == "start"]
 ends = [e for e in events if e["event"] == "end"]
 
 total_duration = sum(
 e["duration"] for e in ends
 )
 
 print(f"{workflow}: {len(starts)} sessions, {total_duration}s total active time")
```

duration isn't the same as tokens, but it's a strong proxy. a session running 8 minutes of active back-and-forth is burning a lot more context than a 45-second one-shot call.

---

**what I found when I actually looked**

the two biggest surprises for me:

1. my "quick question" sessions were fine. the long iterative debugging sessions were the killers. one bad debugging rabbit hole = 3x the tokens of five clean feature builds.

2. I had 3 workflows that loaded the same large context file I never cleaned up. killed those, got meaningful headroom back.

---

**the real fix**

the logging is just the diagnostic layer. the actual fix is building workflow-specific CLAUDE.md files or context files so you're only loading what's relevant.

```
/contexts
 /testing.md # test patterns, framework setup, fixtures
 /deployment.md # infra context, env vars structure
 /architecture.md # system design decisions
CLAUDE.md # 200 lines max. core identity only.
```

then per workflow:
```bash
claude --context ./contexts/testing.md --print "add tests for the new auth flow"
```

your base CLAUDE.md should be your skeleton. everything else is surgical context loading.

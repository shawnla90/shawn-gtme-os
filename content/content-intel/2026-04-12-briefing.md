## 3. [TUTORIAL] the token budget system I use so Claude Code limits never kill a pipeline mid-run

**Thesis:** Hitting limits mid-session is an architecture problem, not a Claude problem. Walk through model routing, context trimming, and session design that keeps production pipelines alive regardless of which model is degraded.
**Target sub:** r/r/ClaudeCode
**Scores:** viral=7 voice=9 gap=7 effort=medium

### REDDIT Version

hitting context limits mid-pipeline is an architecture problem, not a Claude problem

I've had production runs die at step 7 of 12 because the session got too heavy. took me a few burned pipelines to stop blaming the model and start fixing the design. here's the system I run now.

**the core problem**

most people treat Claude Code like a single session that should just... handle everything. the model degrades as context fills. you get slower responses, then truncated outputs, then it just stops mid-tool-call. if you're running automated pipelines this kills your whole run.

the fix is treating token budget as a first-class constraint, same as memory or disk.

---

**1. model routing by task weight**

not every step needs sonnet 4.5. I route by task complexity:

- haiku for classification, formatting, simple lookups
- sonnet for reasoning, code gen, anything requiring multi-step logic
- opus only for architecture decisions or when sonnet has already failed twice on the same task

in practice this means I tag each workflow step with a `model_tier` field. the orchestrator picks the cheapest model that can handle the task. this alone cuts token pressure significantly because you're not burning sonnet context on stuff haiku can handle.

---

**2. context trimming between steps**

the mistake is passing the full conversation history into every subsequent call. you don't need it.

what I do instead: each step gets a structured handoff object, not the raw message history. looks like:

```json
{
 "task_completed": "enriched 847 company records",
 "output_location": "s3://bucket/run_123/enriched.json",
 "errors": [],
 "next_step_context": "filter to companies with >50 employees, output to filtered.json"
}
```

the next agent call starts fresh with just that handoff plus the step instructions. no accumulated history. context stays flat across the whole pipeline no matter how many steps you've run.

for anything that does need memory across steps, I write it to a persistent store (sqlite works fine) and pull only the relevant slice per step rather than dumping everything into context.

---

**3. session checkpointing**

long pipelines need checkpoints. every N steps or every M tokens spent, write state to disk and start a fresh session.

I track this with a simple counter:

```python
CHECKPOINT_EVERY = 5 # steps
TOKEN_THRESHOLD = 80000 # approximate, not exact

if steps_since_checkpoint >= CHECKPOINT_EVERY or estimated_tokens > TOKEN_THRESHOLD:
 write_checkpoint(current_state)
 reset_session()
 steps_since_checkpoint = 0
```

when a session fails mid-run, you restart from the last checkpoint instead of from scratch. this is the difference between losing 2 steps and losing 2 hours.

---

**4. degradation detection**

models don't announce when they're degraded. you have to detect it.

signals I watch for:
- response time spikes (>3x baseline for same task type)
- output length drops significantly on tasks that should be verbose
- structured output stops matching schema on the first try

when I see two of these together I route the next call to a different model tier and log it. usually sonnet is fine, but if there's an outage or capacity issue I want my pipeline to survive it.

---

**5. the session design principle**

stateless steps + stateful store.

each agent call should be able to run with zero knowledge of what came before it, pulling only what it needs from the store. if your pipeline requires a specific call chain to work correctly, you've got hidden state and you'll get weird failures that are hard to debug.

this also makes retries trivial. failed step? rerun it with the same inputs from the store. no need to replay the whole session.

---

none of this is Claude-specific. it's just distributed systems thinking applied to LLM pipelines. treat the model as a stateless compute unit, manage state yourself, and budget tokens the same way you'd budget memory.

```
.claude/
 base.md # global rules, anti-patterns, voice/style
 skills/
 git.md # commit conventions, branch rules
 testing.md # test patterns for this repo
 api.md # http conventions, error handling
 projects/
 lead-pipeline/ # project-specific context

 content-engine/

CLAUDE.md # root . built, not written
```

the root CLAUDE.md isn't hand-written. it's generated by a slash command:

```
/build-context
```

that command looks at:
1. what's in your current working directory
2. which skill files are relevant (based on file types present)
3. the project-specific CLAUDE.md if one exists

and it writes a fresh CLAUDE.md to the root that's actually scoped to what you're doing.

the command itself lives in `.claude/commands/build-context.md` and the prompt is roughly:

```
read the contents of .claude/base.md.
look at the files in the current working directory.
if there's a matching project folder in .claude/projects/, read that CLAUDE.md.
identify which skill files in .claude/skills/ are relevant based on file types and directory structure.
write a new CLAUDE.md to the root that combines: base rules + relevant skill rules + project context.
keep it under 200 lines. remove anything not relevant to this specific project.
```

took about an hour to set up the first time. now I run it at the start of any new work session where I've switched context.

the key insight that changed things: CLAUDE.md is a **prompt**, not a wiki. it should load what's relevant right now, not document everything that ever mattered.

---

- skill files should be declarative, not instructional. "use async/await, never .then() chains" lands better than "you should try to use async/await when possible"
